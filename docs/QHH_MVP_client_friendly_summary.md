# 全国水文模拟系统 MVP 当前建设情况说明（甲方阅读版）

> 适用场景：项目汇报、阶段验收沟通、MVP 上线范围说明  
> 当前 MVP 定位：**QHH/有限流域水文气象展示 + 运维监控**  
> 表述原则：少讲技术细节，多讲“已经能做什么、用户能看到什么、还差什么”。

---

## 1. 项目现在处在什么阶段？

项目已经不只是方案设计，而是进入了**有限流域真实运行和展示阶段**。

目前系统已经完成了基础平台建设，并且以 QHH 流域作为示范流域，跑通了从气象资料获取、模型计算到前端展示的主要流程。

可以简单理解为：

> **系统主体已经搭起来，QHH 示范流域已经能跑起来、算出来、查得到、展示出来。**

当前还不是“全国所有流域全部上线”的阶段，而是先用 QHH/有限流域把 MVP 跑通，再逐步扩展到更多流域。

---

## 2. 这个 MVP 要给用户看什么？

本阶段 MVP 建议只保留两个核心界面。

| MVP 界面 | 面向用户的作用 | 用户能看懂的描述 |
|---|---|---|
| 水文气象展示界面 | 看流域、河段、气象代站和曲线 | 在地图上查看示范流域，点击河段看流量变化，点击气象代站看降雨、气温、湿度、风速等要素变化 |
| 系统运维界面 | 看系统有没有正常运行 | 查看气象数据是否下载成功、模型是否运行成功、结果是否处理完成；失败任务可以重启 |

本 MVP 的水文展示内容是：

> **河段流量曲线**，不是水位曲线。

也就是说，用户看到的是某条河段未来一段时间内流量如何变化，单位通常是立方米每秒。

气象展示内容是：

> **气象代站提取出来的几个关键要素**，包括降雨、气温、湿度、风速、辐射、气压等。

---

## 3. 当前已经完成了哪些工作？

### 3.1 系统基础框架已经完成

系统已经具备后端服务、前端页面、数据库、任务调度、模型运行、结果解析和运维监控等基础能力。

换句话说，系统已经具备了一个业务平台的基本骨架：

- 前端可以打开页面；
- 后端可以提供数据接口；
- 数据库可以保存流域、河段、气象、模型运行结果；
- 模型任务可以提交到计算环境运行；
- 运行日志和任务状态可以被查看；
- 失败任务具备重启能力。

这部分工作为 MVP 上线打好了基础。

---

### 3.2 QHH 示范流域已经跑通过真实流程

目前最重要的成果是：**QHH 示范流域已经完成真实链路跑通**。

已经跑通的流程包括：

```text
气象资料获取
  → 资料格式转换
  → 生成模型需要的气象输入
  → 运行 SHUD 水文模型
  → 解析模型输出
  → 入库保存
  → 前端查询和展示
```

这说明系统已经不是只能看静态页面，而是可以用真实资料驱动模型运行，并把模型结果变成前端可查看的产品。

当前 QHH 已经完成 GFS 和 IFS 多周期运行，并且已有真实运行结果入库。这部分可以作为 MVP 的核心演示内容。

---

### 3.3 河段流量展示能力已经具备

系统已经能够保存并查询河段流量结果。

用户在前端可以围绕“流域—河网—河段”进行查看，点击某个河段后，展示这条河段的流量变化曲线。

当前适合展示的是：

- QHH 示范流域；
- 已经完成模型运行的河段；
- GFS 和 IFS 两类预报来源；
- 河段流量随时间变化的曲线。

需要注意：

> 当前展示的是“流量”，不建议在 MVP 中写成“水位”。  
> 如果后续要展示水位，需要额外增加水位计算或流量—水位换算能力。

---

### 3.4 气象代站展示基础已经具备

系统已经具备气象代站的基础数据结构，也已经能够查询某个模型实际使用的代站列表。

前端也已经有气象产品页面和气象代站页面的基础设计，能够展示：

- 代站列表；
- 代站位置；
- 代站筛选；
- 代站详情；
- 气象要素曲线的页面框架；
- 数据不可用时的提示状态。

这意味着：

> 气象代站“看得到”这件事已经具备基础；  
> 下一步需要把代站曲线真正接到数据库中的真实 forcing 数据上。

MVP 中建议展示以下气象要素：

| 要素 | 业务含义 |
|---|---|
| PRCP | 降雨 |
| TEMP | 气温 |
| RH | 相对湿度 |
| wind | 风速 |
| Rn | 净辐射 |
| Press | 气压 |

这些要素都是水文模型运行所需的重要气象输入。

---

### 3.5 运维监控能力已经具备

系统已经具备运维监控页面和相关后端能力。

运维人员可以查看：

- 当前资料周期是否开始处理；
- 数据下载是否成功；
- 数据转换是否成功；
- 气象输入是否生成完成；
- 水文模型是否运行成功；
- 模型结果是否解析完成；
- 任务日志；
- 队列状态；
- 失败任务。

系统也已经具备失败任务重启接口。

这意味着 MVP 运维界面可以重点展示：

> **系统每一步跑到了哪里，哪里失败了，失败后能不能重启。**

---

## 4. MVP 上线后，甲方能看到什么？

### 4.1 水文气象展示界面

用户进入页面后，可以看到：

1. QHH/示范流域地图；
2. 河网和河段；
3. 气象代站位置；
4. 选择 GFS 或 IFS 数据来源；
5. 点击河段查看流量曲线；
6. 点击气象代站查看气象要素曲线；
7. 查看数据时间、来源和质量状态。

用一句话说：

> 用户可以在地图上看流域，在河段上看流量变化，在代站上看气象输入变化。

---

### 4.2 系统运维界面

运维人员进入页面后，可以看到：

1. 哪个资料周期正在运行；
2. 数据下载是否完成；
3. 数据处理是否完成；
4. 模型是否正在运行；
5. 哪些任务成功、哪些任务失败；
6. 每个任务的日志；
7. 失败任务的重启按钮。

用一句话说：

> 运维人员可以看到系统是否正常运行，并且能处理失败任务。

## 6. MVP 上线前还需要补哪些工作？

为了让 MVP 更稳定、甲方更容易验收，建议重点补齐以下工作。

### 6.1 把气象代站曲线接上真实数据

当前气象代站列表和页面框架已经有了，但还需要把代站对应的气象要素曲线真正接到后端真实数据。

要完成的效果是：

> 点击某个气象代站，可以看到这个代站对应的降雨、气温、湿度、风速、辐射、气压等曲线。

这是水文气象展示界面最重要的补齐项。

---

### 6.2 把 QHH 自动运行纳入统一任务管理

当前 QHH 已经能跑通，但还需要把运行过程更规范地接入统一运维页面。

也就是说，不只是脚本能跑，而是每一步都要在运维界面里看得到：

```text
下载
转换
生成气象输入
模型运行
结果解析
结果发布
```

这样甲方看到的就是一个完整业务系统，而不是后台脚本演示。

---

### 6.3 做一次失败任务重启演练

MVP 里计划展示“失败任务可以重启”。

上线前建议专门做一次演练：

1. 人为制造一个失败任务；
2. 运维页面显示任务失败；
3. 点击重启按钮；
4. 系统重新提交任务；
5. 任务最终成功；
6. 日志和状态都能查到。

这会让甲方非常直观地理解系统的运维价值。

---

### 6.4 简化前端页面

当前前端功能较多，MVP 阶段不建议把所有页面都暴露给甲方。

建议只保留两个入口：

- 水文气象展示；
- 系统运维监控。

其他页面可以作为后续增强能力，不放在 MVP 主导航中。

---

### 6.5 跑一次完整 MVP 验收流程

上线前建议跑一次完整流程：

```text
选择一个 GFS 或 IFS 周期
  → 下载气象资料
  → 生成 QHH 气象输入
  → 运行 QHH 水文模型
  → 解析河段流量结果
  → 前端展示流量曲线
  → 前端展示气象代站曲线
  → 运维页面展示全过程
  → 演示失败任务重启
```

这条流程跑通后，MVP 就具备较完整的演示和试运行条件。

---

## 7. 本阶段不建议承诺的内容

为了避免甲方误解，建议在汇报中明确说明以下内容不属于本 MVP：

| 不建议承诺 | 原因 |
|---|---|
| 全国所有流域全部上线 | 当前先以 QHH/有限流域验证流程，后续逐步扩展 |
| 所有河段水位曲线 | 当前展示的是河段流量，不是水位 |
| CLDAS 真实场正式接入 | 还涉及权限、数据下载和稳定性验证 |
| 全部气象资料源生产级稳定运行 | MVP 先以 GFS/IFS 为主，ERA5/CLDAS 后续完善 |
| 生产级全国瓦片性能 | 全国级性能压测应作为后续生产化工作 |
| 最终生产安全认证体系 | MVP 可做内部试运行，正式生产还需要身份认证、告警、回滚等证明 |

## 9. 当前阶段总体判断

从 MVP 角度看，项目已经具备试运行基础。

更直白地说：

> **能演示主流程，能展示核心结果，能看运行状态；还需要把气象代站曲线和统一运维闭环补得更顺。**

建议将当前 MVP 定位为：

> **有限流域真实试运行版。**

该版本重点证明系统能把气象资料转成水文模型输入，驱动模型计算，并把结果以地图和曲线方式展示出来，同时具备基本运维监控和失败任务处理能力。

后续再逐步扩展为：

```text
更多流域
更多资料源
更完整水文产品
更强运维能力
最终全国生产系统
```
