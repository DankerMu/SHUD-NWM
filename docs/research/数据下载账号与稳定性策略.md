不需要三者都申请账号。建议按下面理解：

|数据源|是否必须申请账号|推荐生产下载方式|稳定性判断|
|---|--:|---|---|
|**GFS**|通常不需要|NOAA Big Data / AWS Open Data、NOMADS、NCEP 产品服务器|最适合作为 MVP 主预报源|
|**IFS Open Data**|通常不需要账号，但要遵守 ECMWF Open Data 使用条款|`ecmwf-opendata` client、ECMWF Open Data Portal、公共云镜像|可作为 GFS 并行预报源，但必须及时镜像|
|**ERA5**|需要 CDS 账号和 API token|Copernicus CDS API / earthkit|适合历史和 analysis，不适合严格近实时|
|**CLDAS**|需要注册/权限|中国气象数据网/气象科学专业知识服务系统|真实场优先源，但权限和自动下载能力要专门跟踪|

你问的是 IFS、GFS、ERA5，我逐个说明。

# 1. GFS：一般不需要申请账号

## 1.1 账号要求

**GFS 实时预报资料一般不需要申请账号。**

项目文档里已经把 GFS 定为 MVP 的 P0 主预报源，说明其权限状态为“开放”，推荐用 GFS 作为未来 0–7 天 forecast forcing 的主数据源。

GFS 推荐使用：

```text
GFS 0.25° pgrb2
00 / 06 / 12 / 18 UTC
f000–f168 覆盖未来 7 天
GRIB2
```

这些在你的气象数据文档里已经写清楚。

---

## 1.2 推荐下载通道

GFS 建议采用多通道冗余。

### 主通道：NOAA Big Data / AWS Open Data

适合生产批量镜像。你文档中已经建议优先使用 **AWS Open Data / NOAA Big Data Program**，并说明它适合云端批量下载和对象存储镜像。

典型路径：

```text
s3://noaa-gfs-bdp-pds/gfs.YYYYMMDD/CC/atmos/gfs.tCCz.pgrb2.0p25.fFFF
```

### 备用通道 1：NOMADS HTTPS filter

适合只下载中国区域、指定变量、指定层，节省带宽。文档中也把 NOMADS filter 列为 GFS 推荐通道之一。

典型路径：

```text
https://nomads.ncep.noaa.gov/pub/data/nccf/com/gfs/prod/gfs.YYYYMMDD/CC/atmos/gfs.tCCz.pgrb2.0p25.fFFF
```

生产上我不建议完全依赖 NOMADS，因为它更适合筛选下载，批量稳定性和限流风险要考虑。

### 备用通道 2：NCEP 产品服务器

作为 NOMADS 或 AWS 异常时的兜底源。

### 历史回补：NCEI

历史归档、长时间回补、漏周期补齐时用。不要把 NCEI 当实时主入口。

---

## 1.3 稳定下载策略

GFS 应该这样设计：

```text
Primary: AWS Open Data / NOAA Big Data
Fallback 1: NOMADS filter
Fallback 2: NCEP 产品服务器
Historical backfill: NCEI
```

下载器不要直接假设某周期已经完整发布，而要做：

```text
1. discover cycle
2. poll file availability
3. build manifest
4. check required forecast hours
5. check required variables
6. download
7. verify file size / checksum / GRIB message count
8. write raw object store
9. mark raw_complete
```

对于系统，GFS adapter 建议配置为：

```yaml
GFS:
  account_required: false
  status: enabled
  cycle_hours_utc: [0, 6, 12, 18]
  forecast_hours:
    start: 0
    end: 168
  preferred_channel: aws_open_data
  fallback_channels:
    - nomads
    - ncep_product_server
    - ncei_archive
  poll_interval_minutes: 10
  max_wait_minutes: 180
  mirror_required: true
```

# 2. IFS Open Data：开放资料通常不需要账号，但必须镜像

## 2.1 账号要求

如果你用的是 **ECMWF IFS Open Data**，通常不需要单独申请 ECMWF 业务账号。  
但必须遵守 ECMWF Open Data 的使用条款和授权要求。

你的文档已经写明 IFS Open Data 是开放资料，权限状态为“开放，需遵守 CC-BY-4.0/ECMWF ToU”，并建议作为 GFS 的并行预报场景接入。

注意这里说的是 **IFS Open Data**。  
如果你想拿 **完整 ECMWF 商业/业务资料、MARS 归档、更多变量、更完整 ensemble 或更长期历史预报归档**，那就不是 Open Data 范围了，通常需要 ECMWF 账号、许可或机构授权。

---

## 2.2 IFS Open Data 的关键限制

IFS Open Data 有两个业务上必须处理的限制：

### 限制 1：06/18 UTC 不一定够未来 7 天

你的文档已经记录：

```text
00/12 UTC：可到 360h
06/18 UTC：到 144h
```

所以 06/18 UTC 不能完整覆盖未来 7 天。

系统必须明确：

```text
IFS 00/12：可作为完整未来 7 天预报
IFS 06/18：仅展示 0–144h，前端标注“可用时效不足 7 天”
```

或者：

```text
IFS 06/18 用最近 00/12 UTC 补齐，但必须保留来源标识
```

不要悄悄补齐，否则曲线会误导用户。

### 限制 2：Open Data 滚动保留时间短

你的文档已经写明 ECMWF Open Data 只保留最近 12 个预报周期，约 2–3 天，因此项目必须及时镜像。

这意味着 IFS 不能“需要时再下载”。必须在周期发布后马上抓取并保存到自己的对象存储。

---

## 2.3 推荐下载通道

推荐使用三层策略：

```text
Primary: ecmwf-opendata Python client, source=ecmwf
Fallback 1: ecmwf-opendata source=aws
Fallback 2: ecmwf-opendata source=azure / google
```

你的文档已经建议 IFS 使用：

```text
ECMWF Open Data Portal
ecmwf-opendata Python client
AWS / Azure / Google Cloud 复制数据
```

并说明公共云复制数据可用于冗余和稳定性提升。

---

## 2.4 稳定下载策略

IFS adapter 建议配置为：

```yaml
IFS_OPEN_DATA:
  account_required: false
  status: enabled
  license_required: true
  license_note: "ECMWF Open Data ToU / CC-BY-4.0"
  cycle_hours_utc: [0, 6, 12, 18]
  lead_time_policy:
    "00": "0-168h for project, source can provide longer"
    "12": "0-168h for project, source can provide longer"
    "06": "0-144h only"
    "18": "0-144h only"
  preferred_client: ecmwf-opendata
  preferred_source: ecmwf
  fallback_sources:
    - aws
    - azure
    - google
  mirror_required: true
  max_wait_minutes: 240
```

系统内建议增加：

```text
ifs_available_end_time
ifs_lead_time_complete_flag
```

前端显示：

```text
IFS 2026-05-03 06 UTC：当前周期仅支持至 +144h
```

# 3. ERA5：需要 CDS 账号和 API token

## 3.1 账号要求

**ERA5 需要 Copernicus Climate Data Store，也就是 CDS 账号和 API token。**

你的文档中已经把 ERA5 的下载渠道写为：

```text
Copernicus Climate Data Store Web Portal
CDS API
earthkit / CDS 工具链
```

并说明 ERA5 用于历史 forcing、长期 analysis、率定和洪水频率样本构建。

所以 ERA5 与 GFS、IFS Open Data 不同，ERA5 自动下载通常需要：

```text
CDS 注册账号
接受数据集条款
配置 API token
使用 cdsapi 或 earthkit 下载
```

---

## 3.2 ERA5 不适合作为严格近实时主源

你的文档已经说明 ERA5 每日更新，但约有 5 天迟滞；ERA5T 后续可能被最终资料替换。

所以 ERA5 的正确角色是：

```text
历史长期 forcing
模型率定
洪水频率样本
历史 analysis
CLDAS 权限未解决时的再分析替代
ERA5T/final 回补
```

不建议用 ERA5 直接承担“最近 0–24 小时真实场”的实时主源。

---

## 3.3 ERA5 稳定下载策略

ERA5 下载最容易遇到的问题是：

```text
CDS 排队慢
请求过大失败
API 限流
部分月份/变量返回慢
ERA5T 后续替换
```

所以不要按“全国全变量全年一次请求”下载。建议拆小任务：

```text
按 source × year × month × variable_group × spatial_bbox 拆分
```

例如：

```text
ERA5_CHINA_2026_04_atmos_basic
ERA5_CHINA_2026_04_radiation
ERA5_CHINA_2026_04_precip
```

推荐变量组：

```text
basic:
  2m_temperature
  2m_dewpoint_temperature
  10m_u_component_of_wind
  10m_v_component_of_wind
  surface_pressure

precip:
  total_precipitation

radiation:
  surface_net_solar_radiation
  surface_net_thermal_radiation
```

ERA5 adapter 建议配置为：

```yaml
ERA5:
  account_required: true
  credential_type: cds_api_token
  status: enabled
  latency_days: 5
  product:
    dataset: reanalysis-era5-single-levels
    format: grib
  request_split:
    by: [year, month, variable_group, area]
  retry_policy:
    max_retries: 5
    backoff_minutes: [10, 30, 60, 180, 360]
  mirror_required: true
  era5t_replacement_policy: true
```

# 4. 三者账号和通道对比

## 4.1 简表

|数据源|账号|实时性|稳定主通道|最重要风险|
|---|---|--:|---|---|
|GFS|一般不需要|高|AWS Open Data / NOAA Big Data|周期文件未完整发布、NOMADS 限流|
|IFS Open Data|一般不需要账号，但需遵守 ToU|高|`ecmwf-opendata` + 公共云镜像|只保留最近约 12 个周期；06/18 时效不足 7 天|
|ERA5|需要 CDS 账号/API token|中低，约 5 天迟滞|CDS API / earthkit|排队、限流、ERA5T/final 替换|
|CLDAS|需要注册/权限|高|CMA 数据平台|批量自动下载权限不确定|

# 5. 系统中不要“边跑边下载”

最重要的架构原则是：

```text
SHUD 运行任务绝对不要直接去外部网站下载 GFS/IFS/ERA5。
```

应该是：

```text
外部资料源
  ↓
DataSource Adapter
  ↓
Raw Downloader
  ↓
对象存储 raw mirror
  ↓
Canonical Converter
  ↓
canonical product
  ↓
Forcing Producer
  ↓
SHUD run
```

你当前文档也已经明确：所有数据源最终都应转换为统一的 `Canonical Meteorological Product`，再由 forcing 生成器映射为 SHUD forcing 文件，不应让 SHUD 运行任务直接读取 GRIB/NetCDF 原始数据。

# 6. 生产稳定下载架构建议

建议新增一个明确的 **Data Mirror Layer**。

```text
data_source_adapter
  ├── discover_cycles()
  ├── build_manifest()
  ├── download_plan()
  ├── download()
  ├── verify()
  └── mirror_to_object_store()
```

对象存储路径建议：

```text
raw/gfs/{cycle_time}/
raw/ifs/{cycle_time}/
raw/era5/{year}/{month}/
raw/cldas/{valid_date}/

canonical/gfs/{cycle_time}/{variable}/
canonical/ifs/{cycle_time}/{variable}/
canonical/era5/{year}/{month}/{variable}/

manifests/{source}/{cycle_or_period}/manifest.json
```

每个 manifest 至少记录：

```json
{
  "source_id": "GFS",
  "cycle_time": "2026-05-03T00:00:00Z",
  "channel": "aws_open_data",
  "files": [
    {
      "uri": "s3://...",
      "local_uri": "s3://nhms/raw/gfs/...",
      "forecast_hour": 0,
      "variables": ["APCP", "TMP", "RH", "UGRD", "VGRD"],
      "size_bytes": 123456789,
      "etag": "...",
      "status": "verified"
    }
  ],
  "complete": true
}
```

# 7. 下载稳定性的关键机制

## 7.1 周期发现不要只靠固定时间

不要写死：

```text
GFS 00 UTC 到 04:00 一定可用
```

而是轮询文件是否完整：

```text
poll cycle
  → check f000/f003/.../f168
  → check required variables
  → if complete then raw_complete
  → else continue until max_wait
```

## 7.2 分级状态

建议每个资料周期有状态：

```text
discovered
downloading
raw_partial
raw_complete
raw_verified
canonical_ready
failed_download
failed_verify
```

你的数据库设计里已经定义了 `met.cycle_status`，包括 `discovered`、`downloading`、`raw_complete`、`canonical_ready`、`failed_download`、`failed_convert` 等状态。

我建议加一个：

```text
raw_partial
```

用于记录部分文件已下载但周期不完整。

## 7.3 多通道 fallback

```text
GFS:
  aws → nomads → ncep → ncei

IFS:
  ecmwf → aws → azure → google

ERA5:
  cdsapi → retry queue → smaller request split → backfill queue
```

## 7.4 永久镜像

IFS Open Data 保留时间短，ERA5 下载慢，GFS 历史回补也不应每次重新下载。因此三者都应该镜像到自己的对象存储。

```text
只要进入业务流程，就必须先落 raw object store。
```

## 7.5 文件级校验

GFS/IFS/ERA5 都要做：

```text
文件大小 > 最小阈值
GRIB message count 合理
变量存在
时间戳一致
forecast hour 完整
checksum / etag 记录
```

## 7.6 变量级校验

不仅要文件存在，还要确认变量存在：

```text
PRCP
TEMP
RH 或可推导 RH 的变量
wind 或 U/V
Rn 或可推导 Rn 的变量
Press
```

SHUD forcing 需要降雨、气温、湿度、风速、辐射和气压等要素，你的气象数据文档也已经明确列出 `.tsd.forc` 和 forcing CSV 中的 `Day, PRCP, TEMP, RH, wind, Rn, Press` 字段。

# 8. 三个 Adapter 的建议实现优先级

## 8.1 GFS Adapter：第一优先级

```text
目标：
  能发现 00/06/12/18 周期
  能下载 f000–f168
  能裁剪中国区域
  能转换 PRCP/TEMP/RH/wind/Rn/Press
  能生成 forcing
```

建议先实现：

```text
GFS 0.25°
NOMADS filter 或 AWS raw download 二选一
优先 f000–f168
步长按源文件可用性读取
```

为了稳定，我建议：

```text
MVP：NOMADS filter 下载中国 bbox + 指定变量
生产：AWS Open Data 全文件或区域后处理 + NOMADS 兜底
```

原因是 NOMADS filter 能节省带宽，适合 MVP；AWS 更适合生产稳定镜像。

---

## 8.2 ERA5 Adapter：第二优先级

```text
目标：
  历史 forcing
  analysis run
  state_snapshot
  洪水频率样本
```

ERA5 不追求分钟级实时，重点是：

```text
请求切分
重试
长期归档
ERA5T/final 标记
```

建议先实现：

```text
按月下载
按变量组下载
中国 bbox
CDS API token 配置
对象存储镜像
```

---

## 8.3 IFS Adapter：第三优先级，但要早做架构预留

IFS 是多 scenario 关键源，但不必放在 MVP 第一阶段。你文档里也建议 GFS 先做 MVP 主链路，IFS 作为并行预报场景接入。

IFS 重点不是账号，而是：

```text
Open Data 保留时间短
06/18 不足 7 天
变量和 step 语义
公共云 fallback
```

# 9. 最推荐的生产下载策略

## 9.1 实时预报下载策略

```text
GFS:
  每天 4 次，00/06/12/18
  主源 AWS / NOAA Big Data
  备用 NOMADS / NCEP
  保存 f000–f168

IFS:
  每天 4 次，00/06/12/18
  主源 ECMWF Open Data
  备用 AWS/Azure/Google mirror
  00/12 保存到 f168
  06/18 保存到 f144，并标记 lead 不足

下载后：
  全部进入 raw object store
  生成 manifest
  再转换 canonical
```

## 9.2 再分析下载策略

```text
ERA5:
  每日检查 available date
  下载 now - 5 days 之前的数据
  按月/变量组/bbox 拆分
  标记 ERA5T/final 版本
  进入 analysis/hindcast/frequency 样本
```

## 9.3 best_available 策略

```text
过去 0–1 天：
  CLDAS 若可用 → CLDAS
  否则 GDAS/GFS analysis/short forecast

过去 1–5 天：
  CLDAS 优先
  ERA5 若尚不可用，用 GDAS/GFS 补

过去 5 天以前：
  ERA5 / ERA5 final

未来 0–7 天：
  GFS scenario
  IFS scenario
  best_available 或 blend 作为派生产品
```

当前系统设计中，`best_available` 已经明确为派生层，不覆盖 GFS/IFS 原始 scenario，并且 v1 采用全域统一选择规则。

# 11. 最终建议

可以这样定决策：

```text
GFS：
  不申请账号，先接入；作为 MVP 主预报源。
  生产采用 AWS/Open Data 主通道 + NOMADS 变量筛选备用。

IFS Open Data：
  不申请业务账号，但遵守 ECMWF Open Data 条款。
  用 ecmwf-opendata client，必须镜像到本系统对象存储。
  06/18 周期只到 144h，要在前端和数据状态中明确标记。

ERA5：
  必须申请/配置 CDS 账号和 API token。
  不作为实时主源，作为历史、analysis、hindcast、洪水频率样本主源。
  必须做请求拆分、重试、ERA5T/final 替换标识。

系统实现：
  外部下载和 SHUD 运行彻底解耦。
  所有资料先进入 raw mirror，再进入 canonical，再进入 forcing。
  任一外部源不可用时，不影响已镜像资料和已发布产品。
```

最关键的一句话是：**GFS 和 IFS Open Data 重点是“多通道镜像和完整性校验”，ERA5 重点是“CDS 账号/API token、请求拆分和迟滞回补”。**
