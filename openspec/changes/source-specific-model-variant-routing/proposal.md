## Why

The direct-grid migration already has upstream capabilities that REGISTER immutable canonical grid snapshots (`canonical-source-grid-registry`) and BUILD immutable source-specific direct-grid model assets (`forcing-mapping-asset-build`), but nothing turns a built variant into a routable, activatable production model. There is no mechanism to register a direct-grid variant as a `core.model_instance` row, no switch to cut a basin over from legacy IDW to direct-grid, no guard stopping a basin from silently regressing to legacy IDW, and no source-scope enforcement at dispatch. This change delivers that routing/activation mechanism — mechanism only — so a later pilot/canary (Change 7) can flip a basin at a cycle boundary. Per docs §16 the switch mechanism (Change 4/4.5/5/8) MUST land before any single-source pilot and MUST activate zero production basins.

## What Changes

- Register a direct-grid variant as a NEW `core.model_instance` row (new `model_id`), grain = basin × `canonical_grid_key` as cross-source dedup: when IFS and GFS share one `grid_signature` they share ONE variant with `resource_profile.direct_grid_forcing.applicable_source_ids=[gfs, IFS]` (never per-source rows); variant identity and idempotency are keyed on the built mapping asset (`model_input_package_id` + `binding_checksum`), so a fix-forward rebuild (M1→M1′) registers as a new row in the same grain while the prior generation is retained; the legacy IDW row is retained forever for calibration/replay and never deleted (INV-1).
- Have the registration step write the `met.met_station` cell-station mirror rows with initial `active_flag=false` — §8.1 forbids the mapping-build stage from touching `met.*`, so the derived mirror belongs to this registration step; each `station_id` has the form `<mapping_asset_identity>::cell:<grid_cell_id>` (from the builder's `assign_station_id_from_mapping_asset_identity`); rows are written in the runtime producer's derived-cache shape (`station_role='direct_grid_cache'` + identity `properties_json`) via a conditional upsert that fails closed on a differing bound identity (§7.4), and the producer's own mirror maintenance is narrowed to preserve the registration-owned `active_flag` (never escalating `false`→`true`) so pre-cutover production cannot activate the mirror.
- Reuse the EXISTING model activation lifecycle for the cutover switch (single-transaction swap, audit, idempotency, concurrency safety) instead of building a new per-`(basin, source)` routing table; every successful transition that changes the dispatch candidate set (`activate`/`switch_version`/`rollback_version`, and `deactivate` of the active model; a standalone supersede-of-active is preflight-blocked today and covered defensively by the same uniform post-commit trigger) re-publishes the scheduler registry manifest to NFS, which the node-22 DB-free scheduler passively consumes.
- Spec-ify and regression-lock the permanent-retirement semantic: activating M1 supersedes M0 in the same transaction and M0 exits the dispatch candidate set — the scheduler consumption face already filters on `lifecycle_state=='active'`; this behavior is promoted to a spec requirement with a regression test, and together with the hard guard forms **permanent retirement** so the basin cannot oscillate between grids.
- Add a lifecycle hard guard: once a basin has direct-grid activation history, any `activate`/`switch_version`/`rollback_version` whose target — the model that would become active (for `rollback_version`, the restored previous model) — is a legacy-mapping model is REFUSED (fail-closed, no override); history is derived from append-only activation audit records so it can neither be erased by deactivate/deprecate nor fabricated by supersede; a direct→direct′ fix-forward swap between direct-grid variants is allowed (docs §11). The guard scopes to production activation only and never blocks offline replay/calibration.
- Enforce source-level isolation at dispatch/staging: fail closed when the requested source is not in the variant's `applicable_source_ids` (INV-4 extension); zero cross-source substitution at the compute layer — one run is single-source end to end, no mid-run splicing, no legacy-package compatibility layer; a source missing data for a cycle means no run for that source/cycle (reason recorded), with cross-source availability still handled by display best-available selection.
- Define the cutover transaction as owned by the lifecycle op with explicit ORDERED pre-activation extension points for Change 5 (state clone) and Change 8 (station flag flip); this change defines the extension-point CONTRACT and empty/no-op implementations only, and does NOT implement clone or flip.
- **Land mechanism only: this change activates zero production basins and performs no cutover on any live basin.**

## Capabilities

### New Capabilities
- `direct-grid-variant-registration`: Register a direct-grid variant as a new `core.model_instance` row at basin × `canonical_grid_key` grain, retain the legacy row, and write the `met.met_station` cell-station mirror with `active_flag=false`.
- `variant-activation-cutover`: Perform the basin cutover by reusing the existing activation lifecycle in one transaction, define the ordered pre-activation extension-point contract, re-publish the scheduler registry manifest, and lock the permanent-retirement (supersede + exit-dispatch) semantic.
- `legacy-reactivation-guard`: Fail closed on any request that would re-activate a legacy-mapping model for a basin with direct-grid activation history, while allowing direct→direct′ fix-forward.
- `source-scoped-dispatch-enforcement`: Fail closed at dispatch/staging when the requested source is outside the variant's `applicable_source_ids`, and forbid cross-source substitution within a single run.

### Modified Capabilities
- `model-version-switch-rollback`: the "Rollback available" guarantee is scoped by the legacy-reactivation guard — a rollback/switch with trustworthy prior state is still refused when the model that would become active is legacy-mapping on a basin with direct-grid activation history.
- `model-operation-preflight`: the enumerated activation-class blocking conditions gain the legacy-reactivation blocker; "Activation preflight passes" no longer holds for a legacy target on a direct-grid-history basin even when basin/river/mesh/package lineage is valid.
- `fixed-station-forcing-production`: the runtime producer's direct-grid `met.met_station` mirror maintenance (DB-plane upsert and file-plane station-inventory handoff) preserves the registration-owned `active_flag` (never escalates `false`→`true`; inserts inactive) instead of hardcoding `true`; mirror activation moves exclusively to the Change 8 cutover flip. Value generation, contract parsing, and the fail-closed collision predicate are unchanged.
- The switch otherwise reuses the existing `model-activation-deactivation` lifecycle without changing its current requirements.

## Impact

- No new per-`(basin, source)` routing table; the switch is the existing `POST /api/v1/models/{id}/lifecycle` activation path (`services/orchestrator` lifecycle op) plus additive guard/extension-point logic.
- New variant-registration surface in `workers/model_registry/` (sibling to `basins_registry_import.py` / `qhh_production_bootstrap.py`) that inserts the `core.model_instance` row and the `met.met_station` mirror rows.
- Narrow producer change in `workers/forcing_producer/` (`store.py:ensure_direct_grid_met_stations` + `file_store.py` station-inventory handoff): mirror `active_flag` preserve-on-update / insert-inactive, replacing the hardcoded `true`; no change to forcing values, parsing, or the collision predicate.
- New ordered pre-activation extension-point contract on the lifecycle cutover op (empty no-op hooks here; Change 5 and Change 8 mount later; hooks skipped on the already-current path).
- Scheduler registry manifest re-publish on every dispatch-set-changing lifecycle transition via the existing `services/orchestrator/scheduler_file_providers.py` publish path → NFS → node-22 DB-free scheduler.
- Source-scope fail-closed check added at the dispatch/staging boundary that pairs a variant with a `(basin, source, cycle)` — the candidate-assembly seam `services/orchestrator/scheduler_candidates.py:build_candidates`, upstream of `scheduler_execution.py:produce_forcing_for_candidates`.
- At most one additive DB migration (numbered from `000046`, the current head being `000045`) if the guard's direct-grid-activation-history marker or the mirror-row → owning-variant linkage cannot be derived from existing schema; no destructive change and no new routing table.
- Depends on `forcing-mapping-asset-build` (the immutable variant + binding + `station_id` helper), `canonical-source-grid-registry` (`canonical_grid_key`, shared-binding eligibility, normalized `applicable_source_ids`), and `cmfd-direct-grid-platform-readiness` (release pin / `z_policy`).
- Legacy CMFD packages, `.sp.att`, `.tsd.forc`, and historical forcing versions remain unchanged (INV-1); the legacy `model_instance` row is retained as immutable lineage (historical products / fingerprint-gated clone source / offline calibration-replay).
- No display/frontend runtime code changes here; the atomic `met.met_station.active_flag` flip and any display-side dynamic model resolution are Change 8.
