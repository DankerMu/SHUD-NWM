## Context

SHUD-NWM runs continuous business forecasts warm-started from prior-cycle SHUD initial-condition snapshots (`strict-warm-start`, `cross-cycle-warm-start-chaining`). Warm-start state lives in `hydro.state_snapshot`, whose unique key is `(model_id, COALESCE(source_id, ''), valid_time)` and whose selection validates `model_package_version` and `model_package_checksum` (db/migrations `000006`, `000028`, `000032`; live audit docs §附录 A, node-27 2026-07-06). Because the key is scoped by `model_id`, warm state is only reusable within the same `model_id`: a cutover from legacy model `M0` to the direct-grid variant `M1` leaves `M1` with no successor state and forces a cold start.

Docs §10 splits cutover visibility by component. node-22 compute is naturally seamless (dual-mode staging path). node-27 display has three concrete break points; this change owns break point 3 (流量展示的水文连续性): river network and hydrologic core are unchanged (INV-2), so the flow layer is structurally seamless, but without a state clone the direct-grid first cycle can only cold-start and the spin-up produces a visible discharge-curve distortion. Docs §Gate G10 fixes the compatibility conditions (core fingerprint equal, state schema equal, checksum valid, valid-time consistent, lineage recorded) and §11 / §11.3 fix the failure-handling model: direct-grid is a one-way channel — no rollback to legacy mapping, only fix-forward.

The gating fingerprint already exists: `workers/mapping_builder/rewrite.py::compute_hydrologic_core_fingerprint` computes the ten-surface Gate G10 `hydrologic_core_fingerprint` (mesh, river, lake, soil, geol, land, calibration, `.sp.att` non-`FORC`, state-schema bytes, solver-config bytes) and `verify_hydrologic_core_fingerprint_equal` is its equality gate. The source of truth for this change is docs/ForcingReplace/CMFD 建模资产向 IFSGFS Direct-Grid 的安全迁移.md §10 断点 3, §11 / §11.3, §12, §Gate G10, and §15 Change 5.

## Goals / Non-Goals

**Goals:**

- Deliver the fingerprint-gated state clone as the state-side mechanism that makes cycle-boundary cutover flow-continuous (§10 断点 3, §Gate G10), executed once per source in the activation's source scope.
- Execute the clone atomically inside the model-activation DB transaction, with whole-transaction rollback on any engaged-clone failure (no "activated-but-not-transferred" state), a pinned applicability predicate (no-op for fresh basins and legacy targets), and an explicit cold-start approval route as the only no-clone commit path.
- Deliver the cloned checkpoint to both warm-start selection planes: the `hydro.state_snapshot` row (DB path) and a checkpoint entry in the scheduler file state index (node-22 DB-free path), published before the registry-manifest re-publish.
- Enforce no-rollback state semantics: no reverse clone onto a legacy-mapping model (Change 4's single classifier); fix-forward `M1→M1′` through the same gate (§11.3).
- Pin replay lineage by interval around the audit-recorded cutover time `t*` (`M0` pre-cutover offline, `M1` at/after, straddling intervals split at `t*`) and prove the Change 4 activation hard guard does not mis-fire on offline replay.
- Record `cloned_from` provenance (source `model_id` + source snapshot identity + gating fingerprint value) as an auditable state lineage.
- Land mechanism only; activate no production basin.

**Non-Goals:**

- Do not define the lifecycle activation transaction or its extension point — that is change `source-specific-model-variant-routing` (Change 4); this change is the extension-point contract's consumer.
- Do not register direct-grid model variants or write `met.met_station` cell-station mirrors (Change 4).
- Do not flip display `active_flag` or touch station-MVT / display surfaces (Change 8).
- Do not perform scientific A/B validation (Change 6, Gate G11).
- Do not execute the cold-start announcement or run the first cold cycle under an approval — the approval audit record is this change's contract; rollout execution is Change 7.
- Do not reimplement or re-derive the `hydrologic_core_fingerprint` — reuse the existing `mapping_builder/rewrite.py` functions (Change 3).
- Do not modify the internals of `strict-warm-start` / `cross-cycle-warm-start-chaining`; the clone shapes the row to satisfy the existing validators, it does not change warm-start requirements.
- Do not activate or migrate any production basin.

## Decisions

### 1. Fingerprint-gated state clone (grill D5)

At cutover the mechanism clones the latest qualified `(M0, source, t*)` snapshot row into `(M1, source, t*)`, once per source in the activation context's source scope (the target variant's normalized `applicable_source_ids` — GFS and IFS states are distinct rows under the unique key, so a dual-source basin gets two clones; one source missing a qualified snapshot fails the whole transaction unless an explicit cold-start approval covers it). **Qualified** is pinned to the Gate G10 conditions: usable, QC-passing, valid `checksum`, `lead_hours=12`, and `valid_time == t*` — the cutover cycle boundary `M1`'s first strict cycle warm-starts from (G10 condition 4); a stale latest snapshot (`valid_time < t*`, e.g. after failed final `M0` cycles) does NOT qualify and routes to the cold-start approval path, because the strict selector would reject it anyway (`valid_time == cycle_time` is hard-enforced).

The physical SHUD state file is NOT copied: hydrologic core and mesh are identical under INV-2, so the same on-NFS state file is naturally legal for `M1`. The clone row's full column disposition is pinned (no lineage-checked column left to guesswork):

- **Preserved verbatim**: `state_uri`, `checksum`, `source_id`, `valid_time`, `cycle_id`, `lead_hours`, `usable_flag`, `original_shud_filename`, and the `M0` producing `run_id` (Decision 3).
- **Overwritten to the target**: `model_id` = `M1`; `model_package_version` = `M1`'s package version (`core.model_instance.model_package_uri` — the strict validators reject on version mismatch: `chain.py::_validate_strict_state_lineage` returns `LINEAGE_PACKAGE_VERSION_MISMATCH`, and the file-index path returns `state_snapshot_index_model_package_version_mismatch`); `model_package_checksum` = `M1`'s package checksum.
- **New**: `state_id` minted via the existing deterministic convention `packages/common/state_manager.py::state_snapshot_id(M1_model_id, valid_time, source_id=…, cycle_id=…, lead_hours=…)` (embeds the new model identity, collision-free against the source row); `created_at` insert default; the three `cloned_from_*`/`clone_gate_fingerprint` provenance columns (Decision 2).

The precondition gate is `verify_hydrologic_core_fingerprint_equal(M0_package, M1_package)` with pinned input authorities: package roots resolved from each model's `core.model_instance.model_package_uri` (NFS object-store path); `category_files` and both `.sp.att` paths taken from the `M1` variant's mapping manifest / mapping evidence package (the same inputs that produced the build-time G4 fingerprint); real platform-level `state_schema_bytes` / `solver_config_bytes` with empty inputs refused fail-closed; and the recomputed `M1` fingerprint cross-checked against the evidence-recorded `hydrologic_core_fingerprint` value, so a symmetric degenerate input can never false-pass the gate. On inequality the clone is refused fail-closed with no row written and the path degrades to the explicit cold-start approval route (§11.3 clause 2; Decision 4 defines the route contract).

- Alternative — copy the physical state file into an `M1`-specific path: rejected. Mesh/core are byte-identical (INV-2), so a copy is redundant IO and a second source of truth that can silently drift from the authoritative NFS file; per the INV-3 spirit the DB row is the index/audit plane, not a second file store.
- Alternative — clone unconditionally without the fingerprint gate: rejected. If the hydrologic core changed, the state vector semantics may differ and warm-starting `M1` from an `M0` state silently corrupts continuity; the fingerprint gate is the fail-closed guarantee that state reuse is only allowed when the core is provably identical (§Gate G10).
- Alternative — treat a stale-but-usable latest snapshot as qualified and rely on `cross-cycle-warm-start-chaining` to bridge: rejected. It would let a cutover commit whose first `M1` strict cycle is still blocked (`valid_time != cycle_time`), reproducing the spin-up break the clone exists to prevent; the stale case is exactly what the explicit cold-start approval route is for.

### 2. `cloned_from` provenance storage: three discrete nullable columns

Provenance is stored as three nullable columns on `hydro.state_snapshot`: `cloned_from_state_id TEXT` (source snapshot identity), `cloned_from_model_id TEXT` (the source `model_id` = `M0`), and `clone_gate_fingerprint TEXT` (the `hydrologic_core_fingerprint` value that gated the clone). All default `NULL`, exactly matching the `000028` warm-start lineage-column style (`source_id`/`cycle_id`/`lead_hours`/`model_package_version`/`model_package_checksum`/`original_shud_filename`), so existing rows and pre-clone producers are untouched.

- Alternative — a single JSONB `clone_provenance` column: rejected. The established schema stores lineage as discrete typed columns; JSONB adds an un-typed, un-indexed blob that diverges from house style, while three scalars are self-describing and directly queryable for audit.
- Migration numbering (concrete collision resolution): the committed head is `000045`. This change needs an unconditional migration for the three provenance columns. The dependency change `source-specific-model-variant-routing` (Change 4) finalized in its design D8 that it introduces **no new migration** — everything it persists fits existing schema, and its `000046` mention is only a reservation for an optional later guard-history index outside its landing scope. This change therefore takes `000046`. In all cases the number is resolved at merge to the next free integer after the highest migration actually on master, never two files at the same number, and `tests/test_migrations.py::test_migration_dependency_order` is rerun after any renumber. Change 8 (`direct-grid-display-cutover`) is display-only and is not expected to add a state migration.

### 3. `run_id` on the clone row (NOT NULL FK handling)

`hydro.state_snapshot.run_id` is `NOT NULL` and references `hydro.hydro_run(run_id)`. The clone row reuses the `M0` producing run's `run_id`: the physical state file was produced by that `M0` run and the clone is an index/audit row pointing at the same `state_uri`, so `run_id` documents physical provenance while `cloned_from_model_id` / `cloned_from_state_id` carry the `M0→M1` transfer lineage. Consumers attributing a state to a model MUST read `model_id` plus `cloned_from_*`, never `run_id` alone — this attribution rule is a spec-level requirement of `fingerprint-gated-state-clone` (with its own scenario), not just a design note, because the clone row is the one place where `snapshot.model_id` (= `M1`) and `hydro_run.model_id` (= `M0`) legitimately diverge.

- Alternative — fabricate a placeholder `M1` `hydro_run`: rejected. No `M1` run has executed at cutover; inventing one pollutes `hydro_run` with a non-run and breaks run-level invariants and joins.

### 4. Same-transaction atomicity, applicability predicate, and the cold-start approval route (grill D6)

The clone is a hook mounted on the ordered pre-activation extension point of Change 4's `variant-activation-cutover` capability (change `source-specific-model-variant-routing`, committed), and it runs in the SAME DB transaction as the activation that marks `M1` active and `M0` `superseded`. The committed Change 4 contract matches the shape this design assumed: hooks run inside the lifecycle transaction before the supersede+activate swap, each receiving the transaction cursor and the activation context (`basin_version_id`, previous active model, target model, source scope); a raising hook aborts the whole transaction with no manifest re-publish; hooks are skipped on the already-current short-circuit; and a mount point is explicitly reserved for this change's state-clone hook.

**Applicability predicate** (which activations engage the clone): the hook no-ops with an audited skip reason unless the activation context carries a previous active model for the scope (`no_previous_active_model` otherwise — a fresh basin routed direct-grid from day one per docs §12 has no warm state to migrate) AND the target model classifies as direct-grid under Change 4's single classifier (`target_not_direct_grid` otherwise — normal legacy operations on the 13 live basins are untouched). The engaged forms are exactly the legacy→direct-grid cutover and the direct→direct′ fix-forward.

**Failure and the cold-start approval route**: any engaged-clone failure — fingerprint-gate refusal, invalid gate inputs, unique-key `(model_id, COALESCE(source_id,''), valid_time)` conflict, or a missing/stale qualified source snapshot — raises and the whole transaction rolls back, so `M1` is never committed active without its successor state and the "activated-but-not-transferred" window §10 断点 3 forbids cannot occur. The refusal surfaces the stable error code `state_clone_cold_start_approval_required` plus an `ops.audit_log` record naming the blocked `(basin_version_id, source_id)` scope and cause. The ONLY way to commit an engaged cutover without a clone row is an explicit cold-start approval input on the activation request (approver identity, reason, covered source ids), recorded in `ops.audit_log` in the same transaction; the approval and the activation result carry the §11.3-clause-3 spin-up-distortion-announcement obligation marker, and executing that announcement / the first cold cycle is Change 7's rollout scope. The hook signature and its order relative to the Change 8 display station-flag-flip hook are owned by Change 4's extension-point contract; this change consumes that contract and declares the dependency as hard blocked-by (no fallback wiring against the bare `model-activation-deactivation` substrate — that would make this change define the transaction it is contractually only consuming).

- Alternative — clone in a follow-up transaction after activation commits: rejected. It re-creates exactly the "activated-but-not-transferred" window; a crash between the two commits leaves `M1` active with no successor state and the first cycle silently cold-starts.
- Alternative — unconditional clone on every activation: rejected. It would block every first-time activation (fresh basins have no `(M0, source, t*)` snapshot by design) and contradict docs §12; the applicability predicate plus the audited skip reason keep the fail-closed property exactly where a warm-state transfer is actually due.

### 5. No-rollback state semantics (grill D7 + §11)

Direct-grid is a one-way channel — **rejecting rollback is a user explicit decision (grill 2026-07-09)**. The clone mechanism forbids any reverse clone onto a legacy-mapping model: there is no code path that clones a direct-grid state back onto a legacy-mapping model. Direction is classified with Change 4's single classifier (pinned by its `legacy-reactivation-guard`): a model is direct-grid iff its `resource_profile.direct_grid_forcing` parses through `workers/forcing_producer/direct_grid_contract.py:load_forcing_mapping_contract_from_manifest` with `forcing_mapping_mode='direct_grid'`; absent/malformed/non-direct classifies as legacy-mapping (fail-closed), so the clone guard and the lifecycle guard can never disagree about a model's class. The guard is enforced at the clone function's own signature (source model row, target model row) as defense-in-depth inside the Change 4 hook — the hook's target is always the activation target, and `activate(M0)` is already refused upstream by Change 4's legacy-reactivation guard, but the clone function itself still refuses a legacy target so no future caller can bypass the semantics. fix-forward `M1→M1′` goes through the same fingerprint gate; because a fix that only touches `FORC`/binding leaves the hydrologic core fingerprint unchanged, the gate passes and the clone proceeds. If the fingerprint differs, the change is not a fix-forward but a new model, and it is routed to the explicit cold-start approval route (§11.3, Decision 4), never to a clone.

### 6. Provenance-pinned replay (grill decision)

Replay/calibration is a lineage-pinned, non-activation operation. The cutover time `t*` is resolved from a recorded authority — the successful direct-grid activation audit record for the basin scope in `ops.audit_log` (Change 4's activation-class audit record), which by construction equals the cloned rows' `valid_time` — never guessed from data availability. The pre-cutover interval runs offline on `M0`, the post-cutover interval uses `M1`, and an interval straddling `t*` is split at `t*` into an `M0` sub-interval (offline) and an `M1` sub-interval, each executed and lineage-recorded separately with no cross-variant splicing (docs §10: 跨 cutover 连续序列不存在，消费方按 variant 分段). Because offline replay never calls lifecycle activation, Change 4's hard guard (which refuses re-activating a legacy-mapping model after direct-grid activation history) MUST NOT intercept offline `M0` replay — the guard is scoped to activation, not to offline runs. The spec carries scenarios proving the guard does not mis-fire on offline replay, so calibration on the legacy driver is never blocked. This change delivers the lineage-pinning resolver as a policy contract (`t*` resolution + interval classification + straddle split plan); the offline replay executor that consumes it is owned by Change 6 (scientific-validation replay) / Change 7 (rehearsal) — there is no existing replay execution path in the repo to modify, and the dispatch plane's `lifecycle_state=='active'` filter already keeps a superseded `M0` out of scheduler dispatch.

### 7. Warm-start delivery to node-22 (fact anchor A-i): DB row plus file-index checkpoint publish

The SHUD state file lives on NFS (`state_uri`), shared between node-27 and node-22 with zero-latency and no rsync; the physical file is unchanged by the clone. The `hydro.state_snapshot` DB row on node-27 is the index/audit plane consumed by the DB-path orchestrator validators. The node-22 DB-free scheduler pass, however, does NOT read the DB: it resolves strict warm-start evidence from the file state snapshot index (`services/orchestrator/scheduler_core.py::_strict_warm_start_for_candidate`, gated on `db_free_required`, calling `packages/common/state_manager.py::FileStateSnapshotIndexRepository.strict_warm_start_evidence` against `NHMS_SCHEDULER_STATE_INDEX`), whose entries are otherwise written only by post-run state save (`packages/common/state_cli.py`) — no DB→index sync process exists. A DB-only clone would therefore leave node-22 blind to `M1`'s successor checkpoint. The cutover consequently publishes one checkpoint entry per cloned `(M1, source, t*)` row into the file index — same identity and lineage as the DB row (`model_id`, `source_id`, `valid_time`, `cycle_id`, `lead_hours`, `state_uri`, `checksum`, `M1` `model_package_version`/`model_package_checksum`) — on the lifecycle post-commit tail, ordered BEFORE Change 4's `publish_scheduler_registry_manifest` re-publish. Failure semantics: if the index publish fails, the manifest re-publish is held back and a recorded blocker demands retry, so node-22 keeps consuming the previous manifest and never observes `M1`-active without `M1`'s successor checkpoint in the index. Because the clone carries the same `state_uri`, the `M1` package version/checksum, and preserved `valid_time`/`cycle_id`/`lead_hours`, the existing strict validators accept it as `M1`'s exact successor on both planes — no file copy and no node-22 DB read.

- Alternative — write the index entry inside the DB transaction hook: rejected. File writes cannot roll back with the transaction; a DB rollback would leave a stale index entry for a never-activated `M1`. Post-commit publish ordered before the manifest re-publish keeps the file plane consistent with what node-22 is allowed to observe.

## Risks / Trade-offs

- Fingerprint gate false-equal from empty state-schema/solver-config bytes → now a spec requirement, not just a caller note: the gate inputs are pinned to authorities (package roots from `model_package_uri`, `category_files`/`.sp.att` from the variant's mapping manifest/evidence package), empty byte inputs are refused fail-closed, and the recomputed `M1` fingerprint is cross-checked against the evidence-recorded value — a symmetric degenerate input cannot survive the cross-check.
- Clone row reuses the `M0` `run_id`, so a naive join over `state_snapshot.run_id` attributes an `M1` state to an `M0` run → the `cloned_from_*` columns make the `M0→M1` lineage explicit; the MUST-level attribution rule (`model_id` + `cloned_from_*`, never `run_id` alone) is a spec requirement with its own scenario.
- A non-atomic clone would leave `M1` active without a successor → the same-transaction hook with whole-transaction rollback (Decision 4) removes the window; the post-commit index publish is ordered before the manifest re-publish with a retry blocker so the file plane cannot expose the window either (Decision 7).
- An unconditional clone hook would block fresh-basin first activations and legacy operations → the applicability predicate (Decision 4) no-ops those with audited skip reasons; the explicit cold-start approval route is the only sanctioned no-clone commit for an engaged cutover.
- A reverse/rollback clone across a core change would silently corrupt continuity → no-rollback semantics reject reverse clone at the clone function signature using Change 4's single classifier, and the fingerprint gate refuses any cross-core clone (Decisions 1, 5).
- The activation hard guard mis-firing on offline `M0` replay would block calibration → replay is a non-activation operation and the guard is activation-scoped; spec scenarios lock the non-interception (Decision 6).
- Migration number collision with a parallel batch-2 change → Change 4's D8 finalized no-migration, so this change takes `000046`; the merge-time next-free-integer rule still applies (Decision 2).
- Consuming a contract owned by another change risks shape drift → the Change 4 extension-point contract is committed (`openspec/changes/source-specific-model-variant-routing/specs/variant-activation-cutover/spec.md`) and matches the shape this design assumes (hook context, in-transaction ordering, reserved state-clone mount point, already-current skip); task 3.1 is hard blocked-by Change 4's extension-point task, with no fallback wiring path.
