## Why

Gate G10 (docs §Gate G10, §10 断点 3) blocks continuous business operation across a legacy→direct-grid cutover: `hydro.state_snapshot`'s unique key is `(model_id, COALESCE(source_id, ''), valid_time)` and warm-start selection validates `model_package_version` and `model_package_checksum`, so when a basin cuts over from legacy model `M0` to the direct-grid variant `M1`, `M1` has no warm-start successor state and is forced to cold-start, producing a visible spin-up flow distortion at the display's discharge curve. This change lands the state-side mechanism — a fingerprint-gated state clone executed atomically with activation — so cycle-boundary cutover carries hydrologic continuity, delivering mechanism only and activating no production basin.

## What Changes

- Add a fingerprint-gated state clone: at cutover, clone the latest qualified `(M0, source, t*)` snapshot row into `(M1, source, t*)` — once per source in the activation's source scope (`applicable_source_ids`); "qualified" means usable, QC-passing, valid checksum, `lead_hours=12`, and `valid_time == t*` (the cutover cycle boundary, Gate G10 condition 4 — a stale latest snapshot does not qualify). The physical SHUD state file is NOT copied (hydrologic core / mesh are identical under INV-2, so the file is naturally legal for `M1`); the clone row's `model_package_version` and `model_package_checksum` are overwritten to the `M1` package version and checksum (both are hard-enforced by the existing strict warm-start validators); `state_uri`, `checksum`, `source_id`, `valid_time`, `cycle_id`, `lead_hours`, `usable_flag`, `original_shud_filename`, and the `M0` producing `run_id` are preserved; the clone row's `state_id` is minted with the existing `state_snapshot_id` convention under the `M1` model identity; `cloned_from` provenance (source `model_id` + source snapshot identity + gating fingerprint value) is recorded. Precondition gate: the `M0` and `M1` `hydrologic_core_fingerprint` MUST be equal, recomputed from pinned input authorities (package roots from `model_package_uri`, `category_files`/`.sp.att` paths from the variant's mapping manifest/evidence package, real platform-level state-schema/solver-config bytes — empty bytes refused) and cross-checked against the evidence-recorded fingerprint; when unequal the clone is refused (fail-closed) and the path degrades to the explicit cold-start approval route defined by `atomic-cutover-transaction`.
- Add DB migration `000046` introducing three nullable lineage columns on `hydro.state_snapshot` (`cloned_from_state_id`, `cloned_from_model_id`, `clone_gate_fingerprint`), each `DEFAULT NULL` so existing rows and pre-clone producers are untouched.
- Add same-transaction atomicity with a pinned applicability predicate: the clone is a hook mounted on the lifecycle activation extension point defined by change `source-specific-model-variant-routing` (Change 4) and runs in the SAME DB transaction as model activation (`M1` active / `M0` superseded). The hook engages only when the activation context carries a previous active model AND the target classifies as direct-grid (Change 4's single classifier); otherwise it no-ops with an audited skip reason, so a fresh basin's first direct-grid activation (docs §12) and normal legacy operations are never blocked. Any failure of an engaged clone rolls back the whole transaction, so no "activated-but-not-transferred" intermediate state can exist; the only way to commit an engaged cutover without a clone row is an explicit cold-start approval input (approver, reason, covered sources) recorded in `ops.audit_log`, with refusals surfacing the stable code `state_clone_cold_start_approval_required`. After commit, the cloned checkpoints are published into the scheduler file state index BEFORE the Change 4 registry-manifest re-publish, so the node-22 DB-free scheduler never observes `M1`-active without its successor checkpoint.
- Add no-rollback state semantics: there is no reverse clone onto a legacy-mapping model (`M1→M0` never happens), with direction classified by Change 4's single direct-grid classifier (`load_forcing_mapping_contract_from_manifest` with `forcing_mapping_mode='direct_grid'`; absent/malformed ⇒ legacy, fail-closed) and enforced at the clone function signature; a fix-forward `M1→M1′` goes through the same fingerprint gate, and a fingerprint-unequal candidate is not a fix-forward but a new model routed to the explicit cold-start approval route (§11.3).
- Add provenance-pinned replay: the cutover time `t*` is resolved from the recorded activation audit authority; the pre-cutover interval replay/calibration runs offline on `M0` (a non-activation operation the Change 4 hard guard MUST NOT intercept), the post-cutover interval uses `M1`, and an interval straddling `t*` is split at `t*` into two lineage-pinned sub-intervals with no cross-variant splicing.

## Capabilities

### New Capabilities

- `fingerprint-gated-state-clone`: Clone the latest qualified `(M0, source, t*)` snapshot into `(M1, source, t*)` per source in the activation scope, behind a `hydrologic_core_fingerprint` equality gate with pinned input authorities, without copying the physical state file, overwriting `model_package_version`/`model_package_checksum` to the `M1` package, minting the clone `state_id` by the existing convention, and recording `cloned_from` provenance; refuse fail-closed on fingerprint inequality, missing/stale qualified sources, or degenerate gate inputs.
- `atomic-cutover-transaction`: Execute the clone as a hook on the Change 4 lifecycle activation extension point within the same DB transaction as activation, engaging only for direct-grid activations with a previous active model, rolling back the whole transaction on any engaged-clone failure, defining the explicit cold-start approval route as the only no-clone commit path, and publishing cloned checkpoints to the scheduler file state index before the registry-manifest re-publish.
- `no-rollback-state-semantics`: Forbid any reverse clone onto a legacy-mapping model (classified by Change 4's single direct-grid classifier, enforced at the clone function signature) and route fingerprint-unequal fix-forward candidates to the explicit cold-start approval route instead of a clone.
- `provenance-pinned-replay`: Pin replay/calibration lineage by interval around the audit-recorded cutover time `t*` (`M0` before, `M1` at/after, straddling intervals split at `t*`) and guarantee the Change 4 activation hard guard does not mis-fire on offline `M0` replay.

### Modified Capabilities

- None.

## Impact

- New migration `db/migrations/000046_state_snapshot_clone_provenance.sql` (number resolved at merge per the coordination rule) adding three nullable `cloned_from_*` / `clone_gate_fingerprint` columns to `hydro.state_snapshot`; the committed head is `000045`. The dependency change `source-specific-model-variant-routing` (Change 4) finalized in its design D8 that it introduces NO migration (`000046` is mentioned there only as a reservation for an optional later guard-history index outside its landing scope), so this change takes `000046`; in all cases the number is resolved at merge to the next free integer after the highest migration on master, and `tests/test_migrations.py::test_migration_dependency_order` enforces strict monotonic ordering and unique filenames and is rerun after any renumber.
- **Depends on change `source-specific-model-variant-routing` (Change 4), which is committed and lands first**, for two contracts this change consumes: (a) the lifecycle activation extension-point contract (`variant-activation-cutover`: ordered pre-activation hooks inside the transaction, hook context = transaction cursor + basin_version_id + previous active model + target model + source scope, a reserved mount point for this change's state-clone hook, hooks skipped on the already-current path), and (b) the `legacy-reactivation-guard` with its single direct-grid classifier (`workers/forcing_producer/direct_grid_contract.py:load_forcing_mapping_contract_from_manifest` with `forcing_mapping_mode='direct_grid'`), which the no-rollback direction classification binds to and the replay guard-scope tests exercise. This change is a consumer of both and defines neither.
- Consumers of the cloned row are the existing `strict-warm-start` and `cross-cycle-warm-start-chaining` warm-start selection paths **on both planes**: the DB-path validators (`services/orchestrator/chain.py::_validate_strict_state_lineage`, `chain_forecast_state.py`) read `hydro.state_snapshot` on node-27, while the node-22 DB-free scheduler pass resolves strict warm-start evidence from the file state snapshot index (`NHMS_SCHEDULER_STATE_INDEX` via `packages/common/state_manager.py::FileStateSnapshotIndexRepository.strict_warm_start_evidence`) — so the cutover also publishes the cloned checkpoint entries into that file index on the post-commit tail, ordered before the Change 4 registry-manifest re-publish. The clone shapes the `(M1, source, t*)` row (same `state_uri`, `M1` `model_package_version`/`model_package_checksum`, preserved `valid_time`/`cycle_id`/`lead_hours`/`source_id`) so the existing strict exact-successor validators accept it for `M1`; the physical state file on NFS is unchanged and node-22 performs no DB read.
- Reuses `workers/mapping_builder/rewrite.py::compute_hydrologic_core_fingerprint` / `verify_hydrologic_core_fingerprint_equal` (the ten-surface Gate G10 fingerprint) as the clone gate; no reimplementation. Gate inputs are pinned: package roots from `core.model_instance.model_package_uri`, `category_files`/`.sp.att` paths from the variant's mapping manifest/evidence package, real platform-level state-schema/solver-config bytes with a fail-closed empty-input refusal, and a cross-check of the recomputed variant fingerprint against the evidence-recorded value.
- No production basin is activated or migrated; this change lands only the state-compatibility mechanism.
- Real-DB oracle is node-27 (`docker exec nhms-db psql -U nhms -d nhms`); the `tests/test_migrations.py` coverage is a static SQL check.
