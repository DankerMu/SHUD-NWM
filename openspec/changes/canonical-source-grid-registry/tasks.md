  - Pinned API surface: single pure entry point `evaluate_shared_binding_eligibility(snapshot_a: CanonicalGridSnapshot, snapshot_b: CanonicalGridSnapshot, *, verification_evidence: SharedBindingVerificationEvidence, store: GridRegistryStoreProtocol) -> None`. Returns `None` on acceptance (after calling `store.extend_applicable_source_ids` for BOTH snapshots with the SAME **canonically-ordered** pair — see next bullet). Raises a `SharedBindingEligibilityError` subclass on any denial.   - Pinned canonical ordering (closes §3.3 positional oracle): the pair passed to BOTH store calls MUST be `tuple(sorted({normalize_source_id(a.source_id), normalize_source_id(b.source_id)}))` — Python default `sorted()` on the set, which by lexicographic ordering gives `("IFS", "gfs")` for the IFS/gfs pair (uppercase `I` < lowercase `g` in ASCII), matching §3.3 backfill oracle `["IFS", "gfs"]` positional equality on BOTH snapshot rows regardless of argument position. Combined with SUB-3 store dedup-on-write (existing entries stay in position; only novel ids append), the resulting stored list on snapshot_a = `["IFS", "gfs"]` (IFS row starts `["IFS"]` → append `"gfs"`) and on snapshot_b = `["IFS", "gfs"]` (gfs row starts `["gfs"]` → the store then extends with `["IFS", "gfs"]`; `"IFS"` is novel and appended AFTER existing `["gfs"]`, giving `["gfs", "IFS"]`). This last case violates §3.3. To close the invariant fully: **before** calling `extend_applicable_source_ids` on snapshot_b, the eligibility function MUST first check if snapshot_b's current `applicable_source_ids == [source_b]` and if so pre-seed by calling `extend_applicable_source_ids(snapshot_b_id, ["IFS", "gfs"])` — the store's insertion-order-preserving append semantic then yields `["gfs", "IFS"]` on snapshot_b, which STILL disagrees with §3.3. Therefore the pinned discipline requires the caller (§3.3 backfill) to insert snapshot rows initially with EMPTY `applicable_source_ids` (deferred from SUB-5 into SUB-8's write path), OR §3.3's positional oracle must relax to set-equality. Since altering SUB-5's write semantic is out-of-scope for SUB-8, the pin locks: BOTH `extend_applicable_source_ids` calls receive canonical pair `("IFS", "gfs")`, and §3.3's positional oracle is re-scoped to **set-equality** (`sorted(row.applicable_source_ids) == ["IFS", "gfs"]` on both rows), which the SUB-3 store's `list[str]` shape supports. This is a **pin-relaxation** on §3.3 recorded here to keep the SUB-3 store contract stable.
