## Why

The direct-grid migration mechanism can now register a direct-grid model variant plus its cell-station mirror (Change `source-specific-model-variant-routing`) and clone a compatible warm state (Change `mapping-variant-state-compatibility`), but nothing flips the *display* from the legacy station set to the new cell-station set. The station-MVT layer queries `met.met_station WHERE basin_version_id=… AND active_flag=true` with no `model_id` filter (`apps/api/routes/hydro_display.py` `_station_source_version`), so a naive dual-track shadow would render old and new stations mixed (and could trip the MVT feature budget). This change delivers the display-side cutover mechanism — an atomic `active_flag` flip on Change 4's lifecycle extension point, bounded old-cycle degradation, active-model dynamic resolution, the station-single-query metadata leak fix (deferred by the readiness §2.3 registration receipt's station-by-id lookup note), and a node-27 synthetic-identity rehearsal receipt — as mechanism only, activating no production basin.

## What Changes

- Add single-track atomic flip: within the cutover DB transaction (mounted on the Change 4 lifecycle extension point), atomically re-point the display station set to exactly the TARGET variant's mirror rows — selected by the target's built mapping-asset identity per the Change 4 registration discriminators (`station_id = "<mapping_asset_identity>::cell:<grid_cell_id>"`, `station_role='direct_grid_cache'`, `properties_json` binding identity, `grid_snapshot_id` FK) — set `active_flag=true`, and set `active_flag=false` for every other `met.met_station` row of the `basin_version` (the legacy set AND every non-target direct-grid generation's mirror, covering the fix-forward direct→direct′ re-flip). The flip hook engages only for direct-grid cutover activations (target classifies direct-grid under Change 4's single classifier AND a previous active model exists), no-oping with an audited skip reason otherwise — mirroring the Change 5 clone-hook classifier — so the 13 live legacy basins' routine lifecycle operations never touch their station sets. The station-MVT query itself is unchanged; dual-track mixed display is prevented by "shadow-period mirror stays `active_flag=false` + single-transaction flip".
- Add bounded historical-cycle display degradation: after cutover, browsing a pre-cutover cycle within the retention window and opening a new cell-station pin renders the existing retention empty state (from `adapt-cycle-picker-retention-window`), not a generic error; the mismatch disappears as the window rotates (mechanically guaranteed by the picker offering only catalog-provided cycles and per-session retained-out marking). "Historical cycle by old variant" is narrowed to product/timeseries data (keyed by `cycle`+`model`, unaffected by the flag — including post-flip `active_flag=false` M0 stations) as a regression guarantee.
- Add active `model_id` dynamic resolution as a spec requirement: no display/frontend surface may pin or cache a `model_id` across a cutover, including MVT tile cache keys. The backend keeps `model_id` an explicit required client-supplied filter (no server-side active-model default or cache); the frontend resolves the live `model_id` from the currently selected latest product at request time. Implementation audits every `model_id` propagation path and closes each one.
- Add the station-single-query `active_flag` leak fix (B4): for an inactive (`active_flag=false`) / evidence-only row, the disk-miss 404 keeps the stable `STATION_FORCING_FILE_NOT_FOUND` code but its `StationForcingFileNotFoundError.details` are desensitized to at most `{station_id}` — disclosing neither the object-store-root-relative storage key (`expected_path`) nor the `(basin_version_id, model_id, source_id, cycle_time)` tuple — closing the enumeration surface recorded in the readiness §2.3 registration receipt. The fix is pinned to 404-details desensitization (NOT lookup filtering, which would break pre-cutover answerability for post-flip inactive M0 stations). Carries a negative test.
- Add a node-27 live rehearsal receipt: provision the rehearsal preconditions under the evidence-only identity (a recorded synthetic previous-active baseline, a Change-4-shaped M1 mirror registered via the Change 4 surface with an object-store-hosted package, a Change 5 cold-start approval, one seeded pre-cutover synthetic cycle/product), run one real cutover transaction (flip → station-MVT before/after diff → popup retention empty-state screenshot → production-scoped 13-basin zero-impact SQL assertions during AND after), then restore end-to-end (deactivate the synthetic model via the Change 4 sys_admin missing-active override, restore all synthetic station flags to `false`, assert the post-restore scheduler manifest carries no synthetic model). The flow-curve cross-cutover continuity receipt is explicitly **DEFERRED** to the pilot's first real cutover and recorded as an absence of evidence, never as a passed verification.

## Capabilities

### New Capabilities
- `station-set-atomic-flip`: Atomically re-point `met.met_station.active_flag` to exactly the target variant's mirror set (all other rows of the `basin_version` → `false`) inside the Change 4 cutover transaction, with a Change-5-style engagement classifier (direct-grid target + previous active model, audited no-op skips otherwise), keeping the station-MVT query unchanged and forbidding any dual-track mixed display — including on direct→direct′ fix-forward re-flips.
- `historical-cycle-display-degradation`: Bound the post-cutover old-cycle mismatch to the retention window by reusing the retention empty state, and keep pre-cutover products/timeseries queryable by `(cycle, model)` key independent of the flag — including for post-flip inactive M0 stations.
- `active-model-dynamic-resolution`: Resolve the active `model_id` dynamically on every display/frontend surface (no cross-cutover pin/cache, including MVT tile cache keys; backend keeps `model_id` an explicit required filter), and close the station-single-query inactive-row metadata leak (B4) by desensitizing the inactive-row disk-miss 404 details.
- `display-cutover-rehearsal-receipt`: Provision, rehearse, and restore a real cutover transaction on the node-27 synthetic identity with production-scoped zero-impact assertions ordered during and after the restore, and explicitly defer the flow-curve continuity receipt to the first real pilot cutover.

### Modified Capabilities
- `object-store-station-series-read`: the disk-miss-404-details requirement is scoped to `active_flag=true` rows; inactive rows keep the stable `STATION_FORCING_FILE_NOT_FOUND` code with details desensitized to at most `{station_id}` (B4). The `STATION_NOT_FOUND` trigger ("a `station_id` not present in `met.met_station`") and successful-read behavior are unchanged. (`met-station-series-api` is NOT modified: its stable `STATION_NOT_FOUND` / `STATION_FORCING_FILE_NOT_FOUND` response codes are untouched — only the reader-owned details payload changes, which `object-store-station-series-read` owns.)

## Impact

- Depends on change `source-specific-model-variant-routing` (Change 4) for the cutover lifecycle extension point where the flag flip mounts, for the activation-context (previous active model / target model) and single direct-grid classifier the flip's engagement and row selection key on, for the mirror-row registration discriminators, and for the guarantee that shadow-period M1 mirrors are registered `active_flag=false`.
- Depends on change `mapping-variant-state-compatibility` (Change 5) for same-transaction integrity: state clone + model activation + station flag flip execute in one DB transaction with no "activated-but-not-transferred / activated-but-not-flipped" intermediate state; the rehearsal additionally uses Change 5's explicit cold-start approval route to traverse the engaged clone hook on the synthetic identity.
- Reuses `adapt-cycle-picker-retention-window` retention empty state; does not rebuild it (no DB fallback, archive read, or synthetic points).
- Display API `apps/api/routes/hydro_display.py` `_station_source_version` (station-MVT source identity / tile cache key) query is unchanged by design — the tile cache key re-derives from the post-flip `active_flag=true` set and self-invalidates on flip.
- Station single-query surface `apps/api/routes/data_sources.py` `/met/stations/{station_id}/series` + `packages/common/object_store_forcing.py` `StationForcingFileNotFoundError` gets the B4 inactive-row 404-details desensitization; `PsycopgStationLookup` keeps resolving inactive rows (no `active_flag` filter), preserving pre-cutover answerability. The deployed `object-store-station-series-read` spec gets the matching MODIFIED delta.
- `met.met_station` schema is unchanged (single-track, no `model` dimension — `db/migrations/000005_met.sql`); no migration is added.
- New tests under `tests/`; frontend `tsc`/`pnpm test` run locally. Display changes are verified via a node-27 live receipt (CLAUDE.md red line), not local ruff.
- No production basin is activated or migrated; the mechanism is inert until change `direct-grid-batch-rollout` (Change 7) references it, and direct-grid is a one-way channel (§11) so the flip is re-applied by fix-forward direct→direct′ (specced and tested as such), never reversed to legacy.
