## Context

SHUD-NWM's direct-grid migration lands its route/state/display mechanisms as separate changes, all "mechanism only, no basin activation" (docs/ForcingReplace §16). Change 4 (`source-specific-model-variant-routing`) registers a new `core.model_instance` variant plus its cell-station mirror rows in `met.met_station` (initial `active_flag=false`, §8.1 forbids writing `met` during the mapping stage), and defines an explicit cutover lifecycle extension point for downstream changes to mount on. Change 5 (`mapping-variant-state-compatibility`) clones the compatible warm state in the *same* DB transaction. What is missing is the display side: nothing flips the visible station set from legacy (M0) to cell stations (M1), and the station-MVT layer query is deliberately model-agnostic.

Live audit (docs/ForcingReplace §附录 A, 2026-07-06, node-27): all 13 basins run legacy IDW, no direct-grid contract exists in any model, and the station-MVT layer queries `met.met_station WHERE basin_version_id=… AND active_flag=true` with **no** `model_id` filter (`apps/api/routes/hydro_display.py` `_station_source_version`, :582-614). Because a mapping variant shares the same `basin_version_id`, once a direct-grid station mirror lands, that one query would return both the legacy and the cell station sets — mixed display, and possibly an MVT feature-budget overflow. The气象代站 timeseries endpoint (`packages/common/object_store_forcing.py` `read_station_forcing_csv`, :342) resolves CSVs via the station inventory's `forcing_filename`, not an `X<lon>Y<lat>` filename pattern, so the new naming is natively compatible — but the storage path contains a `model_id` segment, so any surface that pins an old `model_id` across cutover would read the wrong path.

The source of truth is docs/ForcingReplace/CMFD 建模资产向 IFSGFS Direct-Grid 的安全迁移.md, primarily §10 (switch-visibility boundary, the three break-points), §11 (one-way channel / fix-forward), §15 Change 8 (coverage list), and 附录 A (live audit facts). This change is scoped to the three node-27 display break-points of §10 plus the station-single-query leak defer recorded in the readiness §2.3 registration receipt (`openspec/changes/archive/2026-07-10-cmfd-direct-grid-platform-readiness/evidence/db-registration-2.3.node-27.pass.log`, station-by-id lookup note: "does NOT filter active_flag or station_role … tightening the station-by-id filter is deferred to a separate change"). Note: the source design doc (§15 Change 8, line 1076) labels this record "readiness §2.4 N1" — that label is an erratum; the record lives in the §2.3 registration receipt, and no "N1" label exists in the readiness archive. This change anchors to the real artifact.

## Goals / Non-Goals

**Goals:**

- Flip the visible station set atomically to exactly the target variant's mirror inside the Change 4 cutover transaction, with the station-MVT query itself unchanged, covering both legacy→direct and direct→direct′ fix-forward.
- Structurally forbid dual-track mixed display: shadow-period mirrors stay `active_flag=false`; the flip is single-transaction; at no instant is the MVT set the union of two station sets (legacy vs mirror, or two mirror generations).
- Bound the post-cutover old-cycle mismatch to the retention window by reusing the `adapt-cycle-picker-retention-window` empty state, and keep pre-cutover products/timeseries answerable by `(cycle, model)` key — including for post-flip inactive M0 stations.
- Make active `model_id` resolution dynamic everywhere (no cross-cutover pin/cache, including MVT tile cache keys) and close the B4 single-query inactive-row metadata leak with a negative test, without breaking historical answerability.
- Produce one node-27 real-transaction rehearsal receipt on the synthetic identity — provisioned, executed, and restored end-to-end — proving zero production impact with production-scoped assertions, and explicitly defer the flow-curve continuity receipt.

**Non-Goals:**

- Do not register or activate/migrate any production basin; this change only lands the display-side cutover mechanism (activation is Change 7).
- Do not change the station-MVT source query (`_station_source_version`) to filter by `model_id` — that is explicitly out of design (single-track flip replaces it).
- Do not define the lifecycle extension point (Change 4) or the state-clone same-transaction integrity (Change 5); this change mounts on them.
- Do not rebuild the retention empty state (`adapt-cycle-picker-retention-window` owns it) or add DB fallback / archive reads / synthetic station-series points.
- Do not add a `model` dimension or cycle-awareness to `met.met_station` (see Decision 2, rejected alternative).
- Do not certify the flow-curve cross-cutover continuity here (DEFERRED to pilot; recorded as absence of evidence, not a gap).
- Do not implement the fresh-basin (no previous active model) display bring-up — the flip hook no-ops there with an audited skip; first-display of a day-one direct-grid basin is owned by the rollout change (Change 7).

## Decisions

### 1. Single-track atomic flip on the Change 4 lifecycle extension point, keyed to the target variant's identity

The cutover re-points the display set in one operation inside the activation transaction, mounted on the extension point Change 4 defines (hooks receive the transaction cursor and the activation context: `basin_version_id`, previous active model, target model, source scope). `met.met_station` has no `model` column (`db/migrations/000005_met.sql`:47-57), so row selection is bound to the discriminators Change 4 persists at registration:

- **Activate**: exactly the TARGET variant's mirror rows — matched from the activation context's target model via its `resource_profile.direct_grid_forcing` built mapping-asset identity (`model_input_package_id` + `binding_checksum`), i.e. the rows with `station_id = "<mapping_asset_identity>::cell:<grid_cell_id>"`, `station_role='direct_grid_cache'`, matching `properties_json` binding-identity fields, and the `grid_snapshot_id` FK of the target's registered snapshot.
- **Deactivate**: every other `met.met_station` row of the `basin_version` — legacy rows (`grid_snapshot_id IS NULL`) AND every non-target direct-grid generation's mirror rows. This is what makes the direct→direct′ fix-forward re-flip correct: at M1→M1′ both generations' mirrors coexist (Change 4 allows multiple built generations per grain, possibly bound to the same snapshot), and a naive "legacy off / cell on" reading would commit M1 ∪ M1′ mixed display; keying the activate set to the target's mapping-asset identity and defaulting everything else to `false` yields exactly one generation active.

**Engagement**: the flip is a self-gating pre-activation hook, mirroring Change 5's clone-hook classifier — it engages only when (a) the target model classifies as direct-grid under Change 4's single classifier (`resource_profile.direct_grid_forcing` parses via `workers/forcing_producer/direct_grid_contract.py:load_forcing_mapping_contract_from_manifest` with `forcing_mapping_mode='direct_grid'`, fail-closed) and (b) the activation context carries a previous active model. A legacy-target `activate`/`switch_version`/`rollback_version` (the 13 production basins' routine operations) no-ops with the audited skip reason `target_not_direct_grid` and touches no station row — without this gate, a routine legacy switch would blank the basin's station layer ("deactivate legacy stations" with zero mirrors to activate). A first activation with no previous active model no-ops with `no_previous_active_model` (there is no display set to cut over from; fresh-basin bring-up is Change 7's). The already-current short-circuit never invokes hooks (Change 4's contract).

The station-MVT query keeps filtering only by `basin_version_id + active_flag=true`. This is the whole "single-track" idea: at any committed instant exactly one station set is active, so the layer never mixes sets and the MVT feature budget is never fed a union.

Alternative considered: a per-`(basin, source)` display routing table or a `model_id` filter on the MVT query. Rejected — it duplicates the model-activation lifecycle, adds a second source of truth for "which stations are live", and re-introduces the mixed-display window during the switch. The `active_flag` flip already is the switch.

### 2. Rejected: add a `model` dimension + cycle-awareness to `met.met_station`

The tempting "make old cycles show old stations" design is to key `met.met_station` (and the MVT query) by `model_id` and resolve the station set per browsed cycle. **Rejected (grill D8-D9).** It pays *permanent* schema + query + tile-cache-key complexity to cosmetically paper over a mismatch that is *retention-bounded* — the old cycle rotates out of the retention window within days and the mismatch disappears on its own. The single-track schema (`db/migrations/000005_met.sql`:47-57, no `model` column) is a confirmed design invariant, and the bounded degradation is already handled by the retention empty state (Decision 3). Paying a forever cost for a limited-time visual artifact is exactly the trade grill rejected.

### 3. Bounded degradation via the existing retention empty state

After cutover, a user browsing a pre-cutover cycle within the retention window who opens a *new* cell-station pin will request a station-series file that does not exist for that old cycle → `STATION_FORCING_FILE_NOT_FOUND`. The `adapt-cycle-picker-retention-window` capability already renders a retention-specific empty state for exactly that response (no chart, no generic error, no DB fallback). This change *reuses* it — the new-pin-on-old-cycle case is a natural hit of that empty state. Boundedness is asserted through its frontend-observable mechanics, not through an untestable temporal claim: the issue-time picker offers only catalog-provided cycles (it never synthesizes or persists pre-cutover options) and the retained-out marking is per-cycle, per-session state with no persistence — so once upstream retention rotation stops offering pre-cutover cycles, no mismatch surface remains. The "historical cycle by old variant" concern is narrowed to the data layer: pre-cutover flow products and station/forcing timeseries stay queryable by their `(cycle, model)` key (immutable old-variant assets), independent of the `active_flag` flip — explicitly including post-flip `active_flag=false` M0 stations — and this change locks that with regression scenarios.

### 4. Active `model_id` resolved dynamically; MVT tile cache key is not model-pinned

Because the storage path carries a `model_id` segment (Decision Context), any surface that pins/caches an old `model_id` reads the wrong path after cutover. Spec requirement: no display/frontend surface pins or caches a `model_id` across a cutover. There is deliberately no server-side active-model resolution on the station-series path: `model_id` is a required client-supplied filter (`MISSING_REQUIRED_FILTER` when absent, `apps/api/routes/data_sources.py`:141 + `packages/common/object_store_forcing.py`), and the requirement pins that it stays that way (no server-side default, no server-side cache); the dynamic resolution lives at the caller — the frontend popup derives `model_id` from the currently selected latest product (`apps/frontend/src/components/map/M11StationForcingPopup.tsx` `stationSeriesProductIdentity`). The MVT tile cache key derives from the station *source identity* (`_station_source_version`, the `active_flag=true` set), so it self-invalidates on flip — the query is intentionally not keyed to a `model_id`, and keeping it that way is a decision, not an oversight. Implementation is an audit of every `model_id` propagation path (display API query params, tile cache keys, frontend request builders) with a per-point close-out. A per-request *explicit* old `model_id` for a historical cycle stays legal (immutable old asset); that is historical resolution, not a pinned live default.

### 5. B4 leak fix: desensitize the inactive-row disk-miss 404 details (NOT lookup filtering)

`PsycopgStationLookup._lookup_with_cursor` (`packages/common/object_store_forcing.py`:180-204) looks up a station by `station_id` with no `active_flag` filter, and `StationForcingFileNotFoundError.details` (:120-127) echoes `expected_path` — the object-store-root-relative storage key (`_object_store_relative_path` strips the `OBJECT_STORE_ROOT` prefix, :243-249; the deployed `object-store-station-series-read` spec calls it "an object-store-relative `expected_path`") — plus `(basin_version_id, model_id, source_id, cycle_time)`. So an authenticated user can query an inactive / evidence-only `station_id` and enumerate its storage key + identity tuple. The readiness §2.3 registration receipt recorded this and deferred the fix here (`db-registration-2.3.node-27.pass.log`: "does NOT filter active_flag or station_role … tightening the station-by-id filter is deferred to a separate change if needed"; the source design doc's "§2.4 N1" label for it is an erratum — see Context).

The fix form is pinned: keep the stable `STATION_FORCING_FILE_NOT_FOUND` code for inactive rows but desensitize its `details` to at most `{station_id}`. The superficially equivalent alternative — filtering `active_flag` inside the lookup — is NOT equivalent and is rejected: after the flip, M0 legacy stations are `active_flag=false` while their pre-cutover CSVs still exist within retention; the same `PsycopgStationLookup` is the single read path for `/met/stations/{station_id}/series`, so filtering would turn every post-flip historical M0 read into `STATION_NOT_FOUND` even when the file exists — breaking this change's own `historical-cycle-display-degradation` guarantee (pre-cutover timeseries by old `model_id` stay answerable, independent of the flip) and contradicting the deployed `STATION_NOT_FOUND` trigger ("a `station_id` not present in `met.met_station`"). It must also not over-filter the other way: an *active* station legitimately missing a cycle's file keeps its full `STATION_FORCING_FILE_NOT_FOUND` details (positive test); an inactive row whose file exists keeps serving its series (cross-requirement test). The deployed `object-store-station-series-read` disk-miss-details requirement gets a matching MODIFIED delta. Note readiness treated the evidence *identifiers* as non-secret by design; the closure is specifically the storage-key + tuple enumeration surface — row-existence observability of non-secret IDs stays out of scope.

### 6. Live receipt = provisioned, executed, and restored synthetic-identity rehearsal on node-27; flow-curve deferred

The receipt runs a *real* cutover transaction against the readiness synthetic identity's `basin_version` (`basin__evidence_cmfd_p02_synth__v1`, registered on node-27 DB per `openspec/changes/archive/2026-07-10-cmfd-direct-grid-platform-readiness/evidence/register-synth-p02.sql` + `db-registration-2.3.node-27.pass.log`). The archived registration alone cannot traverse the real path — its `model_package_uri` is a GitHub tree URL (Change 4 preflight blocks it), its `synth-station-001..003` rows are legacy-shaped `forcing_proxy` rows (a correct flip selects zero of them as a target mirror), it has no previous active model (the flip and clone hooks would no-op), and it has no browsable cycle/product — so the rehearsal is specced in three phases:

1. **Provision (recorded, evidence-only)**: a synthetic previous-active baseline model made `active` by recorded SQL (a recorded bypass — the rehearsed lifecycle op is the cutover, not the baseline), with the legacy-shaped `synth-station-*` rows provisioned `active_flag=true` as its display set; a Change-4-shaped synthetic M1 registered through the Change 4 registration surface (conforming mirror shape, object-store-hosted package + rereadable checksum so preflight passes); a Change 5 explicit cold-start approval covering the synthetic sources, so the engaged clone hook traverses its real sanctioned approved-skip path (the positive clone body stays owned by Change 5's verification); one seeded pre-cutover synthetic cycle (ready forecast `hydro.hydro_run` + latest-product row bound to the baseline), which is what makes the synthetic basin discoverable in the `/` basin selector (`GET /api/v1/basins` `has_display_product` resolves basins from ready forecast runs) and the popup path reachable.
2. **Execute**: one real Change 4 `activate` with mounted hooks: flip → station-MVT before/after diff (before = source-version string over the provisioned legacy set, after = string over exactly the M1 mirror) → popup retention empty-state screenshot on the live host (browse the seeded pre-cutover cycle, open a new cell-station pin) → production-scoped SQL assertions **during the committed window**: the 13 production basins' `active_flag` unchanged and active `core.model_instance` count excluding `model__evidence%` = 13, with the transient global count of 14 recorded explicitly as the expected committed-rehearsal state.
3. **Restore**: deactivate the synthetic model via the Change 4 `deactivate` lifecycle op committed with the sys_admin missing-active override (the flip hook is pre-activation-only, so station flags are then restored to `false` by recorded SQL); re-run the production-scoped assertions **after restore** (global count back to 13); assert the post-restore re-published scheduler registry manifest contains no `model__evidence%` model and no `hydro.hydro_run` row was created for any synthetic model during the window (the rehearsal is timed between scheduler cycle boundaries). The receipt enumerates exercised real-path steps versus recorded bypasses, so its claims match what actually ran.

Per CLAUDE.md this display receipt is produced on node-27 (`https://test.nwm.ac.cn` public + `docker exec nhms-db psql`), not faked with local ruff. The flow-curve cross-cutover continuity receipt is **DEFERRED**: no production basin is activated here, so no real cross-cutover flow curve exists to sample; the deferral is written into the receipt as a recorded absence of evidence bound to backfill at the pilot's first real cutover, so it is never misread as verified.

## Risks / Trade-offs

- **Risk: flip half-applies (previous set off, target set still off → empty display; or target on before activation → mixed).** → Mount the flip on the Change 4 extension point so it runs inside the single activation+clone transaction; any step failing rolls the whole transaction back (no intermediate state). Regression scenario locks "at most one set active at any committed instant".
- **Risk: fix-forward direct→direct′ commits both mirror generations active (mixed display).** → Row selection is keyed to the TARGET's mapping-asset identity with all other rows of the `basin_version` forced `false`; a dedicated fix-forward scenario + test locks "only the target generation ends active".
- **Risk: the flip hook fires on a routine legacy lifecycle op and blanks a production basin's station layer.** → Engagement classifier mirrors Change 5 (direct-grid target + previous active model, else audited no-op skip); a negative test asserts the 13 production basins' station sets are untouched by legacy-target operations.
- **Risk: shadow-period M1 mirror is accidentally `active_flag=true` → mixed display.** → Change 4 registration guarantees `false`; this change adds a regression scenario asserting shadow mirrors are not displayed and are `active_flag=false` until the flip.
- **Risk: frontend/display caches an old `model_id` → wrong storage path after cutover.** → Active-model dynamic-resolution spec requirement + audit of every propagation path including the MVT tile cache key; backend regression locks "no server-side model_id default/cache".
- **Risk: B4 fix breaks historical reads or legitimate active-station 404s.** → Fix form pinned to inactive-row 404-details desensitization; positive test keeps active-station `STATION_FORCING_FILE_NOT_FOUND` details; cross-requirement test keeps post-flip inactive M0 stations serving existing pre-cutover files; negative test proves the storage key + tuple are no longer enumerable.
- **Risk: rehearsal touches production or leaves residue.** → Synthetic identity only; production-scoped SQL assertions (excluding `model__evidence%`) run during AND after; restore deactivates the synthetic model (sys_admin missing-active override) and returns all synthetic station flags to `false`; post-restore manifest assertion + no-synthetic-run assertion prove the scheduler plane is clean. The transient global active count of 14 and the append-only audit rows are recorded expected states, not violations.
- **Risk: the DEFERRED flow-curve receipt is misread as verified.** → The receipt records it as an explicit absence of evidence bound to pilot backfill; certification declares it a recorded deferral, not a gap.
- **Risk: old-cycle degradation is perceived as a regression.** → It is bounded by the retention window and rendered as the existing retention empty state, not an error; the "visible change" (station count ~5×, id/filename churn, §10) is the migration's intent and only occurs at real cutover (Change 7), not here.

## Migration Plan

1. Land the flip mechanism (target-identity row selection + engagement classifier) on the Change 4 lifecycle extension point, the B4 single-query 404-desensitization fix with its `object-store-station-series-read` MODIFIED delta, and the active-model dynamic resolution, with no basin activation.
2. Add unit/integration coverage per capability, including the negative scenarios (no mixed display incl. fix-forward, legacy-target no-op, bounded degradation, no cross-cutover pin, inactive-row leak closed, post-flip historical answerability).
3. Produce the node-27 synthetic-identity rehearsal receipt in three recorded phases — provision (baseline + Change-4-shaped M1 + cold-start approval + seeded cycle) → execute (flip → MVT diff → retention empty-state screenshot → during-window production-scoped assertions) → restore (deactivate via sys_admin missing-active override + flag restore + post-restore assertions + clean-manifest/no-dispatch proof) — and record the flow-curve continuity receipt as DEFERRED-to-pilot.
4. The mechanism is inert until Change 7 (batch rollout) references it. There is no rollback surface here: direct-grid is a one-way channel (§11); the flip is re-applied by fix-forward direct→direct′ (covered by its own scenario and test), never reversed to legacy-mapping.

## Open Questions

- None blocking. The B4 fix form is pinned to inactive-row 404-details desensitization; the once-considered "filter `active_flag` at the lookup" alternative is rejected as non-equivalent (it breaks post-flip historical answerability for M0 stations and the deployed `STATION_NOT_FOUND` trigger — see Decision 5), resolving the collision the source design doc left open between §10 break-point 1 (single-query tightening) and break-point 2 (historical cycles stay answerable by old `model_id`).
