## Context

The direct-grid migration's upstream mechanics already exist:

- `canonical-source-grid-registry` (archived) registers immutable grid snapshots, derives the source-agnostic `canonical_grid_key`, and decides GFS/IFS shared-binding eligibility by `grid_signature` equality.
- `forcing-mapping-asset-build` (archived) builds an immutable direct-grid variant (rewritten `.sp.att`, binding + manifest matching the parser contract, evidence package) and assigns `station_id` via `assign_station_id_from_mapping_asset_identity` (`workers/mapping_builder/binding.py:1501`, format `f"{mapping_asset_identity}::cell:{grid_cell_id}"`, separator `STATION_ID_SEPARATOR = "::cell:"` at `binding.py:264`).
- `direct-grid-forcing` (archived) is the runtime consumer: `workers/forcing_producer/direct_grid_contract.py` parses `resource_profile.direct_grid_forcing`, `_applicable_source_ids` (`direct_grid_contract.py:246-293`) already fails closed with `DirectGridContractError("Direct-grid contract does not apply to the current source.")` when a caller-supplied `source_id` is not in the normalized `applicable_source_ids` list.
- The model lifecycle already exists: `POST /api/v1/models/{model_id}/lifecycle` (`apps/api/routes/models.py:617`) delegates to `PsycopgModelRegistryStore.model_lifecycle_operation` (`packages/common/model_registry.py:1457-1648`), which runs a single `with self._transaction() as cursor:` (line 1494), locks the basin_version scope `FOR UPDATE` (1511-1525), computes preflight blockers via `_build_model_operation_preflight` (2090), and on `activate`/`switch_version` supersedes the prior active model and activates the target in the same cursor (`_apply_model_lifecycle_transition` 2256-2303, low-level writer `_update_model_lifecycle_state` 2305-2344). `lifecycle_state` (`db/migrations/000022_model_asset_lifecycle.sql`) is `TEXT DEFAULT 'inactive'` with a CHECK over `('inactive','active','deprecated','superseded')` (22-24) and a partial unique index enforcing one active model per `basin_version_id` (61-63).
- The node-22 scheduler is DB-free and passively consumes an NFS manifest published by `publish_scheduler_registry_manifest` (`services/orchestrator/scheduler_file_providers.py:539`); its dispatch-candidate list already filters `str(row.get("lifecycle_state") or "active") == "active"` (`scheduler_file_providers.py:119`, consumed via `scheduler_models.py:41` `fetch_active_model_details` → `list_models(active=True)`).

What is missing is the seam between "a built variant" and "a routed, activatable production model": there is no code that registers a direct-grid variant as a `core.model_instance` row, no cutover switch, no guard against silently regressing a cut-over basin back to legacy IDW, and no source-scope fail-closed at dispatch. This change delivers that seam — **mechanism only**. Per docs §16 the switch mechanism (Change 4/4.5/5/8) MUST land before any single-source pilot and MUST activate zero production basins.

A foundational product decision constrains the whole design: **direct-grid is a one-way channel — rejecting rollback to legacy IDW is an explicit user decision (grill 2026-07-09; docs §11).** Once a basin is cut over, the only emergency path is fix-forward (direct→direct′); a hard lifecycle guard refuses any legacy re-activation, fail-closed, with no override.

Source of truth: docs §10 (cutover visibility boundary), §11 (fail-handling one-way channel), §15 Change 4 (coverage checklist, rewritten per grill), §16 (Go/No-Go + implementation order), and the grill decisions D1–D3/D6 + A-i/A-vii pinned below.

## Goals / Non-Goals

**Goals:**

- Register a built direct-grid variant as a NEW `core.model_instance` row at basin × `canonical_grid_key` grain, retaining the legacy IDW row immutably.
- Write the `met.met_station` cell-station mirror rows at the registration step with `active_flag=false` (§8.1 keeps `met.*` writes out of the mapping-build stage).
- Perform the cutover by reusing the existing single-transaction activation lifecycle, re-publishing the scheduler registry manifest on success.
- Promote the permanent-retirement semantic (supersede M0 in the same transaction + exit dispatch candidates) to a spec requirement with a regression test.
- Add a fail-closed lifecycle guard refusing legacy re-activation once a basin has direct-grid activation history (no override), while allowing direct→direct′ fix-forward.
- Fail closed at dispatch/staging when the requested source is outside the variant's `applicable_source_ids` (INV-4 extension) and forbid cross-source substitution within a run.
- Define the cutover transaction's ordered pre-activation extension-point contract with empty/no-op hooks, so Change 5 (state clone) and Change 8 (station flag flip) can mount later.

**Non-Goals:**

- Do NOT activate, cut over, or migrate any production basin; this change lands inert mechanism only (docs §16).
- Do NOT implement the state clone (Change 5) or the station `active_flag` flip / display resolution (Change 8); only the empty extension-point hooks are defined here.
- Do NOT build variants, bindings, or evidence (`forcing-mapping-asset-build`), and do NOT register grid snapshots or change eligibility (`canonical-source-grid-registry`).
- Do NOT perform scientific A/B validation (Change 6) or pilot/canary rollout (Change 7).
- Do NOT create a new per-`(basin, source)` routing table; the switch is the existing lifecycle path.
- Do NOT change the runtime producer's value generation, the contract parser, or the scheduler's manifest transport. The ONLY producer-side change in scope is the `met.met_station` mirror `active_flag` ownership rule (D2: preserve the registration-owned flag, never escalate `false`→`true`; insert inactive) — the producer's forcing values, contract parsing, and collision predicate are untouched; beyond that only the source-scope precondition and the manifest re-publish trigger are added.
- Do NOT write `met.interp_weight`, `met.forcing_version`, cycle `.tsd.forc`, or station CSVs — those remain runtime-producer outputs (§8.1/§8.3).

## Decisions

### D1. Variant = a new `core.model_instance` row at basin × `canonical_grid_key` grain (grill D1)

A direct-grid variant is registered as a NEW `model_id` row in `core.model_instance` (`db/migrations/000004_core.sql:71-85`), sharing the baseline's `basin_version_id`, `river_network_version_id`, `mesh_version_id`, and hydrologic-core fingerprint, and carrying the direct-grid contract in `resource_profile.direct_grid_forcing`. Grain is basin × `canonical_grid_key`, and the grain rule is **cross-source dedup, not lifetime uniqueness**: when IFS and GFS resolve to one `canonical_grid_key` (equal `grid_signature`, per `source-shared-binding-eligibility`), they share ONE variant whose `resource_profile.direct_grid_forcing.applicable_source_ids` lists both normalized ids (`[gfs, IFS]`) — never per-source rows. Variant identity and registration idempotency within the grain are keyed on the **built mapping asset identity** (`model_input_package_id` + `binding_checksum` from the §7.2 manifest): re-registering the same built asset returns the existing row; registering a DIFFERENT built asset for the same `(basin_version_id, canonical_grid_key)` (a §11.2 fix-forward rebuild, M1→M1′) inserts a NEW row, so successive fix-forward generations coexist in one grain with at most one `active` (the existing partial unique index on `basin_version_id`). The `canonical_grid_key` itself is copied verbatim from the registered `met.canonical_grid_snapshot` row (resolved via the built manifest's `grid_signature`/`grid_id` or an explicit `grid_snapshot_id` input — `canonical_grid_key` is not a §7.2 manifest field and is never recomputed by the registration surface) and persisted at `resource_profile.canonical_grid_key`, top-level alongside — not inside — the parser-validated `direct_grid_forcing` block, so the grain/idempotency lookup is a derived query over existing columns. The legacy IDW `model_instance` row is retained forever — never deleted, never mutated (INV-1) — as calibration/replay lineage. Registration follows the `basins_registry_import.py:_ensure_model_instance` precedent (INSERT with `active_flag=false` literal, `lifecycle_state` left to its `'inactive'` default), so a freshly registered variant is inert until a separate activation.

- **Alternative — a new per-`(basin, source)` routing table** mapping each source to a required model id: rejected. It duplicates the lifecycle's active-model authority, creates a second source of truth for "what runs", and the existing partial unique index already enforces one active model per `basin_version_id`. Reuse of `resource_profile.direct_grid_forcing` (the parser's authoritative location) keeps a single authority.
- **Alternative — one variant per source even when signatures match**: rejected by grill D1 and §2.2 — it produces 13×2 geometrically identical bindings. Shared `canonical_grid_key` means one variant; per-source divergence (forcing package, warm state, scheduler routing) stays independent because those are keyed on `source_id` downstream, not on the variant row.

### D2. Registration writes the `met.met_station` mirror with `active_flag=false` (grill D2, A-i)

The `met.met_station` cell-station mirror is a derived cache (INV-3). §8.1 forbids the mapping-build stage from writing any `met.*` row, so the mirror is written by THIS registration step, not by the builder. Each mirror row uses `station_id = f"{mapping_asset_identity}::cell:{grid_cell_id}"` (the builder's `assign_station_id_from_mapping_asset_identity` output), `basin_version_id` = the shared basin version, `geom = ST_SetSRID(ST_MakePoint(lon, lat), 4490)` (the SRID-4490 CGCS2000 mirror of the WGS84 binding coordinate), and — critically — **`active_flag=false` set explicitly**, because `met.met_station.active_flag` defaults to `true` (`db/migrations/000005_met.sql:54`). The row MUST set the `grid_snapshot_id` FK column added by `db/migrations/000043_canonical_grid_snapshot.sql` to bind the mirror to its canonical snapshot; that FK gives Change 8 a clean discriminator (legacy stations have `grid_snapshot_id IS NULL`) without a new column.

**Row shape = the producer's derived-cache contract.** These rows already have a second writer: every direct-grid production run calls `ensure_direct_grid_met_stations` (`workers/forcing_producer/producer.py:518` → `workers/forcing_producer/store.py:334-430`), whose conditional `ON CONFLICT ... DO UPDATE` only matches rows with `station_role='direct_grid_cache'` and a `properties_json` carrying the derived-cache identity fields (`derived_cache:true`, `forcing_mapping_mode`, `binding_checksum`, `model_input_package_id`, `grid_signature`, `contract_grid_id`, `grid_id`), failing closed otherwise. Registration therefore writes the SAME shape — `station_role='direct_grid_cache'` (not the `'forcing_proxy'` default) plus the full identity `properties_json` from `store.py:347-379` including `binding_uri`, `sp_att_path`, `sp_att_checksum`, `grid_cell_id`, `shud_forcing_index`, and `forcing_filename` (the object-store station-series read path resolves by it) — so a later production run reconciles the registration-owned rows instead of tripping its collision error.

**Collision policy is fail-closed, not clobber.** The write is a CONDITIONAL upsert with an affected-row-count check, mirroring the producer's policy (`store.py:400-429`) and docs §7.4: an existing `station_id` whose bound identity (binding checksum / model package / grid signature / grid snapshot / coordinates) differs fails registration closed with no row mutated. The plain unconditional `qhh_production_bootstrap.py:_seed_station_rows` `ON CONFLICT (station_id) DO UPDATE` precedent is reused only for statement shape, NOT for its unconditional overwrite semantics.

**Flag ownership is reconciled with the producer (scoped producer change).** Today `ensure_direct_grid_met_stations` hardcodes `active_flag=true` on insert and `DO UPDATE SET ... active_flag = true`, and the DB-free file plane emits `"active_flag": True` per station (`workers/forcing_producer/file_store.py:759`) into the `station_inventory.json` → `met.met_station` handoff — either would silently defeat the registration-owned `false` during any pre-cutover (shadow) production run. This change therefore scopes a narrow producer change (see the `fixed-station-forcing-production` delta): both planes preserve an existing row's `active_flag` on update (never escalate `false`→`true`) and insert fresh rows inactive; mirror activation belongs exclusively to Change 8's atomic flip. The producer's value generation, parsing, and collision predicate are unchanged.

- **Why `active_flag=false` (grill A-i, docs §10 breakpoint 1):** station-MVT queries `met.met_station WHERE basin_version_id=… AND active_flag=true` (`apps/api/routes/hydro_display.py`), NOT by `model_id` (there is no `model_id` column on `met.met_station`). If the mirror landed active during the shadow window, the display layer would show both the legacy and the direct-grid station sets for one basin (mixed display, and possible MVT feature-budget overflow). Keeping the mirror `active_flag=false` preserves a single visible track until Change 8's atomic flip.
- **Alternative — write the mirror at activation time instead of registration**: rejected. The mirror is a large per-cell insert; doing it inside the cutover transaction bloats the critical activation path and couples asset-shape to the switch. Registration is the correct owner (assets exist before activation); the cutover only flips `active_flag`.
- **Alternative — never persist the mirror, resolve stations from the binding at read time**: rejected. The display/MVT read path consumes `met.met_station` rows; re-architecting that is Change 8's non-goal here, and INV-3 already sanctions a derived mirror.
- Registration writes ONLY the `met.met_station` mirror; it does NOT write `met.interp_weight`, `met.forcing_version`, or any cycle forcing — those are runtime-producer outputs (§8.1/§8.3). Direct-grid uses one-cell `weight=1.0`, so no IDW weight rows are produced.

### D3. Cutover reuses the existing activation lifecycle; no new routing table (grill D3)

The cutover switch is exactly `POST /api/v1/models/{model_id}/lifecycle` `operation=activate` targeting the direct-grid variant, executed by `PsycopgModelRegistryStore.model_lifecycle_operation` (`model_registry.py:1457-1648`). That path already provides single-transaction swap, `FOR UPDATE` concurrency safety, idempotent/already-current handling, and an `ops.audit_log` row in the same transaction. On successful activation the lifecycle op re-publishes the scheduler registry manifest via `publish_scheduler_registry_manifest` (`scheduler_file_providers.py:539`) → NFS → the node-22 DB-free scheduler passively consumes the new active model (`scheduler_models.py:41` `list_models(active=True)`). No new switch, table, or transport is introduced.

- **Alternative — a dedicated cutover endpoint/state machine**: rejected. It would re-implement the lifecycle's transaction, audit, idempotency, and concurrency guarantees. The migration's job is routing, not a new lifecycle.
- The manifest re-publish is made an explicit post-commit step of the activation so the compute plane cannot serve a stale active model after a cutover; the transport itself (NFS write) is unchanged.

### D4. Permanent retirement: supersede + exit-dispatch, spec-ified and regression-locked (grill D3, docs §15/§11)

Activating M1 supersedes M0 in the same transaction (existing behavior at `_apply_model_lifecycle_transition:2271-2274`), and the superseded M0 exits the dispatch candidate set because the scheduler consumption face filters `lifecycle_state == "active"` (`scheduler_file_providers.py:119`). This change promotes that already-present behavior to a spec requirement and locks it with a regression test, so a future refactor of the scheduler filter or the transition writer cannot silently re-admit a superseded grid. Together with the D5 guard this constitutes **permanent retirement**: a basin cannot oscillate between grids, which is what would re-introduce the cross-grid representativeness error the migration exists to remove. M0 is retired, not destroyed — it remains immutable lineage for historical products, the fingerprint-gated clone source (Change 5), and offline calibration-replay.

- **Alternative — a new explicit "retired" lifecycle state**: rejected. `superseded` already means "was active, replaced", the CHECK constraint enumerates exactly four states, and the scheduler already excludes non-`active`. Adding a state would touch the CHECK, every projection, and the scheduler filter for no behavioral gain.

### D5. Legacy-reactivation hard guard: fail-closed, no override (grill, docs §11)

Once a basin has direct-grid activation history, any `activate`/`switch_version`/`rollback_version` whose TARGET is a legacy-mapping model is REFUSED. This is implemented as a NEW preflight blocker in the existing `_build_model_operation_preflight` (`model_registry.py:2090`), inside the activation-class path already gated at `activation_class_operation = operation in {"activate","switch_version","rollback_version"}` (line 2111). The three predicate terms are pinned as follows, all derivable from existing schema — there is no `model_class` column and no new migration:

- **"Target"** ⟺ the model that would become `active` as a result of the operation. For `activate`/`switch_version` that is the addressed model; for `rollback_version` it is the RESTORED model — the preflight's own `restored_model = previous_model if operation == "rollback_version" else model` (`model_registry.py:2105`) — because rollback requires the addressed `{model_id}` to be the current active model and re-activates `previous_model` (`model_registry.py:2294-2302`). Classifying the addressed model would make the guard vacuous on the canonical rollback vector.
- **"Legacy-mapping" / "direct-grid" classification** ⟺ ONE classifier everywhere: a model is direct-grid iff its `resource_profile.direct_grid_forcing` parses successfully through `load_forcing_mapping_contract_from_manifest` (`workers/forcing_producer/direct_grid_contract.py`) with `forcing_mapping_mode='direct_grid'`; absent, malformed, or non-direct contracts classify as legacy-mapping (fail-closed). A present-but-invalid direct-grid contract is thus REFUSED as a target on a direct-history basin, with a blocker code distinguishing "invalid direct-grid contract" from "legacy reactivation". The same classifier is used for the refused target, the permitted fix-forward target, and the history predicate — no field-equality shortcut that could diverge from the parser.
- **"Basin has direct-grid activation history"** ⟺ an APPEND-ONLY predicate, NOT current `lifecycle_state`: `EXISTS` an `ops.audit_log` record of a successful activation-class transition (`entity_type='model_instance'`, `action IN ('models.activate','models.switch_version','models.rollback_version')`, `details->>'outcome' IN ('allowed','rollback')`, `details->>'basin_version_id'` = the scope — the exact record shape the rollback-history query already reads at `model_registry.py:1996-2009`) whose resulting-active model (`details->'updated_model'->>'model_id'`, joined back to `core.model_instance`, which is never deleted per D4/INV-1) classifies as direct-grid; OR the scope's currently active model classifies as direct-grid (belt-and-braces for any activation that predates audit coverage). A `lifecycle_state IN ('active','superseded')` predicate was rejected because current state is mutable in both directions: `deactivate` (legal from `active`, writes `'inactive'`, `model_registry.py:2275-2278` — and §11.2 step 4 makes deactivate-after-cutover a designed pause lever, reachable via the sys_admin missing-active override) followed by `deprecate` would ERASE history and re-admit legacy activation, while `supersede` (legal from `'inactive'`, `model_registry.py:2280-2286`) on a never-activated variant would FABRICATE history and block a live legacy basin. The audit-based predicate survives deactivate/deprecate (audit rows are append-only) and ignores supersede-from-inactive (no activation record, not currently active). An `inactive` registered-but-never-activated direct-grid variant creates no activation audit record and does NOT arm the guard — which is exactly why THIS mechanism-only change, which activates nothing, leaves the guard inert for all 13 live basins.

The guard is fail-closed with **no override flag** — unlike the deactivate override path (`OVERRIDE_REQUIRES_SYS_ADMIN`, `model_registry.py:2137-2166`). **Rejecting rollback to legacy is an explicit user decision (grill 2026-07-09).** A direct→direct′ swap (target is itself a direct-grid variant) is allowed, so fix-forward works. The guard scopes to production activation ops only; offline replay/calibration uses the legacy package directly (not via the lifecycle activate op) and is never blocked (grill decision 5, docs §11.2/§12).

- **Alternative — a soft warning or a sys_admin override**: rejected by the one-way-channel decision. An override re-opens the oscillation the design forbids; the emergency path is fix-forward, not rollback.
- **Alternative — keying history on `lifecycle_state IN ('active','superseded')`**: rejected as shown above — erasable via deactivate/deprecate, fabricable via supersede-from-inactive; the one-way channel must not be escapable with ordinary supported operations.
- **Alternative — a persisted `direct_grid_activated_at` marker column (new migration)**: rejected as unnecessary. The predicate is derivable from append-only `ops.audit_log` + immutable `core.model_instance` rows; see D8. Migration `000046` stays reserved if the history query ever needs an index or the audit join proves insufficient in practice.

### D6. Source-scope fail-closed + zero cross-source substitution (grill D6, INV-4 extension)

Dispatch/staging that selects a variant for a `(basin, source, cycle)` MUST fail closed when the requested `source_id` is not in the variant's `applicable_source_ids`. This extends INV-4 (explicit direct-grid never falls back to IDW) to the source axis and reuses the existing membership check: the parser's `_applicable_source_ids` (`direct_grid_contract.py:246-293`) already raises `DirectGridContractError("Direct-grid contract does not apply to the current source.")` when a caller passes a `source_id` outside the list — so the dispatch layer MUST invoke the contract with the requested source and treat that error as a hard refusal, never a silent fallback. Compute-layer isolation is absolute: one run is single-source end to end, no mid-run splicing of another source's cells, no legacy-package compatibility layer. A source missing data for a cycle means no run for that `(source, cycle)` with the reason recorded; cross-source availability is display's `best-available-selection` concern, never a compute substitution (§2.3, §10 accepted-visible-change).

- **Alternative — fall back to any applicable source when the requested one is missing data**: rejected. It re-introduces mixed-grid runs (§2.3) and defeats the per-source warm-state and evidence model. Availability is resolved at display, not compute.
- **Alternative — enforce source scope only inside the producer**: rejected. INV-5 requires the production route to explicitly demand the right variant BEFORE forcing production; the check therefore belongs at the dispatch/staging boundary that picks the variant, in addition to the parser's own check.

### D7. Cutover transaction owns ordered pre-activation extension points (contract + empty hooks) (grill A-vii, docs §15/§10)

The cutover transaction (`model_lifecycle_operation`) is the owner of an ORDERED sequence of pre-activation hooks that run INSIDE the same DB transaction, before the atomic supersede+activate swap. This change defines only the hook CONTRACT and registers EMPTY/no-op hooks; it implements neither clone nor flip:

- Each hook receives the transaction cursor and an activation context (basin_version_id, previous active model, target model, source scope) and either succeeds or raises.
- A hook raising aborts the whole transaction (fail-closed) — there is no "activated but hooks not run" intermediate state. This is the invariant Change 5 and Change 8 depend on.
- Hooks run in a declared, stable order. Change 5 mounts a fingerprint-gated state-clone hook; Change 8 mounts a `met.met_station.active_flag` flip hook. Both mount onto this contract without re-opening the lifecycle transaction.

- **Alternative — post-activation hooks (after commit)**: rejected. A post-commit clone/flip creates exactly the "activated but state not transferred / stations not flipped" window docs §10 breakpoint 3 forbids. Pre-activation, same-transaction hooks make the cutover atomic.
- **Alternative — a separate orchestrator that clones/flips after the lifecycle call**: rejected for the same atomicity reason and because it splits the transaction owner.

### D8. No new DB migration (derivable from existing schema); 000046 reserved if an optional index is later needed

Everything this change persists fits existing columns: the variant is a `core.model_instance` row (`resource_profile` JSONB holds the `direct_grid_forcing` contract incl. `applicable_source_ids`, plus the top-level `resource_profile.canonical_grid_key` copied from the registered snapshot for the D1 grain/idempotency lookup; `lifecycle_state`/`active_flag` already exist), and the mirror is a `met.met_station` row (`station_role`, `active_flag`, `properties_json`, and the `grid_snapshot_id` FK added by 000043). The D5 guard predicate is a derived query over append-only `ops.audit_log` activation records joined to immutable `core.model_instance` rows (plus the currently-active check), not a stored flag and not mutable `lifecycle_state`. Therefore this change introduces **no new migration**. The current head is `000045`; IF an implementer later measures the guard's history query as hot enough to warrant a partial index (e.g. on direct-grid `model_instance` rows per `basin_version_id`), that index is an additive migration numbered from `000046` — never destructive, never a routing table, and DB assertions MUST NOT reference `000042` (a dropped-index migration with no direct-grid content).

- **Alternative — add a `direct_grid_activated_at` / `variant_kind` column now**: rejected as YAGNI. The distinction is already expressible; a premature column adds a backfill and a second authority for a fact the contract already carries.

## Risks / Trade-offs

| Risk | Mitigation |
| --- | --- |
| The mechanism accidentally activates or disturbs a live basin | Every task is "mechanism, not activation"; the variant registers as `lifecycle_state='inactive'`; no task calls `operation=activate` on a production basin; a regression test asserts the 13 live basins' active model is unchanged after registration. |
| Mirror lands `active_flag=true` (default is true) → shadow-window mixed display | D2 sets `active_flag=false` explicitly; a negative test proves a registered variant's mirror rows are all `active_flag=false` and never appear in the `basin_version_id + active_flag=true` MVT query. |
| A pre-cutover production run force-escalates the mirror to `active_flag=true` (producer hardcodes `true` on both DB and file planes) | D2 scopes the producer flag-ownership change (`fixed-station-forcing-production` delta): preserve-on-update, insert-inactive; a regression test proves a production run against an inactive variant leaves every mirror row `active_flag=false`. |
| Registration mirror rows don't match the producer's collision predicate → first production run fails closed; or an unconditional upsert clobbers a foreign station_id | D2 pins the row shape to the producer's derived-cache contract (`station_role='direct_grid_cache'` + identity `properties_json`), and the registration write is a conditional upsert with affected-count check that fails closed on a differing bound identity (docs §7.4). |
| Guard is armed for a basin that only registered (never activated) a variant, breaking the 13 live basins | D5 history requires a successful activation audit record (or a currently active direct-grid model); registration writes neither; a test proves registration alone leaves legacy activation permitted, and a supersede of a never-activated variant does not arm the guard. |
| Guard history erased by deactivate/deprecate after cutover → legacy re-activation admitted | D5 keys history on append-only `ops.audit_log` activation records, not mutable `lifecycle_state`; a regression test deactivates (and deprecates) the cut-over direct-grid variant and asserts legacy activation is STILL refused. |
| Guard blocks legitimate direct→direct′ fix-forward, or is vacuous on rollback | The predicate refuses only when the model that would BECOME active is legacy-mapping (for `rollback_version` that is the restored `previous_model`); a direct-grid target is allowed; positive test covers the fix-forward swap and a negative test covers rollback-to-legacy. |
| Fix-forward M1′ unregistrable because the grain already has M1 | D1 keys identity/idempotency on the built asset (`model_input_package_id` + `binding_checksum`); a successor asset registers as a new row in the same grain; a test proves M1 is retained and M1′ gets a new `model_id`. |
| Source-scope check silently falls back to IDW or another source | The check reuses the parser's fail-closed membership error and is placed at the dispatch/staging boundary (INV-4/INV-5); negative tests prove an out-of-scope source refuses to run with no fallback. |
| A pre-activation hook failure leaves a half-applied cutover | Hooks run inside the lifecycle transaction; a raising hook rolls the whole transaction back; a test injects a raising hook and asserts no state change and no manifest re-publish. |
| Scheduler filter refactor silently re-admits a superseded grid | D4 promotes the `lifecycle_state=='active'` filter to a spec requirement with a regression test pinning `scheduler_file_providers.py:119`. |
| Manifest not re-published after a dispatch-set-changing transition → node-22 serves stale candidates | Manifest re-publish is an explicit post-commit step of EVERY successful transition that changes the dispatch candidate set (`activate`/`switch_version`/`rollback_version`/`deactivate`/`supersede` of the active model — deactivate is the §11.2 step-4 pause lever); per-operation tests assert `publish_scheduler_registry_manifest` is invoked exactly once on success and not on a blocked, rolled-back, or already-current operation. |

## Migration Plan
